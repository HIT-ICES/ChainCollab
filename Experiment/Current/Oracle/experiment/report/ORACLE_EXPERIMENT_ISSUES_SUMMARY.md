# Oracle 实验问题总结

本文件用于记录 Oracle 侧实验在调整过程中的关键问题、处理方式与当前可用口径，便于论文中统一说明实验边界。

## 1. 已保留的原始结果

Oracle 侧所有原始结果均已保留，没有覆盖旧数据。当前同一实验通常保留两类文件：

- 原始结果文件
- 修正后的结果文件（例如 warmup 版）

常见文件包括：

- `experiment/report/compute-10-scenarios-report.json`
- `experiment/report/compute-10-scenarios-report.warmup.json`
- `experiment/report/COMPUTE_TASK_EXPERIMENT_REPORT.md`
- `experiment/report/COMPUTE_TASK_EXPERIMENT_REPORT.warmup.md`

## 2. 主要问题与处理

### 2.1 计算任务实验的首样本偏差

问题：
- 计算任务实验中，首个可比较样本的链下提交成本明显偏高。
- 原因是第一次写入状态时会触发额外的 storage 初始化开销，导致首样本不代表稳态成本。

处理：
- 为计算实验增加 warm-up 模式，先提交一次预热任务，再进入正式测量。
- 保留原始结果，同时新增 warmup 版结果，便于对照。

当前口径：
- 原始版本适合展示“首次调用”成本。
- warmup 版本适合展示“稳态执行”成本，论文主文建议优先采用这一版。

### 2.2 数据聚合实验的仿真口径

问题：
- 数据聚合实验不是直接使用原始工业在线流，而是基于公开数据集构造多节点观测与异常扰动场景。
- 如果不说明清楚，容易被误解为真实传感器网络原位采集。

处理：
- 在实验报告中明确写明：该实验为仿真实验，采用公开数据集并模拟多节点观测。
- 当前主表采用“按场景汇总”的形式，不再强调每个数据集的细分差异。

当前口径：
- 主要关注方法在不同数据质量条件下的适配性。
- 结果分析更适合围绕“哪类聚合方法更适合较干净数据、哪类更适合存在异常/缺失的数据”展开。

### 2.3 安全实验的简化

问题：
- 安全实验如果包含过多攻击向量，报告解释成本高，且主结论容易分散。

处理：
- 将安全实验收敛为三类典型攻击：
  - 越权提交
  - 错误签名
  - 重放

当前口径：
- 只保留这三类攻击作为主实验结论。
- 其余更细粒度攻击可在实现层保留，但不再作为主报告指标。

### 2.4 拜占庭容错实验的定位

问题：
- 该实验更像是鲁棒性 benchmark，而不是严格意义上的形式化容错证明。

处理：
- 在报告中将其定位为“鲁棒性/容错效果验证”。
- 避免把它表述成底层协议级安全证明。

## 3. 当前建议的论文口径

### 3.1 计算任务

- 建议主文采用 warmup 后的稳态结果。
- 原始结果保留为补充材料或附录。

### 3.2 数据聚合

- 建议明确写成“基于公开数据集的仿真实验”。
- 主文重点讨论方法适配性，不强调更复杂的数据来源解释。

### 3.3 安全验证

- 建议只保留越权、错误签名、重放三类攻击。
- 主结论聚焦“拦截有效且不影响可用性”。

## 4. 结论性说明

Oracle 侧目前没有阻断实验推进的关键问题。  
主要需要在论文里统一说明两点：

1. 计算任务实验采用 warm-up 结果作为稳态口径；
2. 数据与安全实验都属于可复现实验/仿真实验，而不是对真实工业链路的线上部署验证。
