# 第X章 实验设计与结果分析

## 1. 实验目标
本文围绕 Oracle 机制验证以下三个问题：

1. 数据型任务中，工业数据聚合链路在噪声/漂移/缺失场景下是否具备稳定可用性（成本仅作附带参考）。
2. 计算型任务中，链上直接计算与链下计算回传两种路径的成本差异如何，何种任务适合下链。
3. 白名单 + 结果签名机制能否有效阻断典型攻击，同时保持任务可用性与结果完整性。

## 3. 实验一：工业数据聚合有效性

### 3.1 实验设置
- 实验目标：验证本文数据任务聚合方案在“较干净数据”与“问题数据（异常/缺失）”两类条件下的可用性与聚合质量。
- 数据集：采用 UCI `Room Occupancy Estimation`（ID: 864），使用其中两组多传感器同指标数据，分别为 `Temp` 与 `Light`，每组采样 90 轮。

| 数据集 | UCI/来源 | 指标列 | 链接 |
| --- | --- | --- | --- |
| Room Occupancy Estimation | UCI 864 | Temp | https://archive.ics.uci.edu/dataset/864/room+occupancy+estimation |
| Room Occupancy Estimation | UCI 864 | Light | https://archive.ics.uci.edu/dataset/864/room+occupancy+estimation |

- 实验流程：在本地以太坊测试链部署预言机合约，启动多个预言机节点并将节点观测映射为同一指标的不同传感器读数；随后注册数据任务并分别在三类场景下执行：
  - 轻微扰动场景
  - 单点异常场景
  - 缺失数据场景
- 聚合方法（中文）：算术平均、中位数、加权平均、截尾平均。
- 评估指标：`MAPE`（聚合质量）与 `Gas`（链上开销，单位 `gas`）。结果表按“场景”展示，为跨 Temp/Light 两组指标取平均后的结果。

### 3.2 结果汇总（可用性主表）
| 聚合方法 | 轻微扰动场景-MAPE | 轻微扰动场景-Cost(gas) | 单点异常场景-MAPE | 单点异常场景-Cost(gas) | 缺失数据场景-MAPE | 缺失数据场景-Cost(gas) |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| 算术平均（MEAN） | 0.10% | 848,193.172 | 10.61% | 848,097.467 | 5.53% | 821,055.706 |
| 中位数（MEDIAN） | 0.09% | 870,395.111 | 0.40% | 870,716.744 | 0.38% | 843,327.061 |
| 加权平均（WEIGHTED\_MEAN） | 0.10% | 974,039.300 | 10.61% | 974,132.389 | 5.53% | 945,992.856 |
| 截尾平均（TRIMMED\_MEAN） | 0.09% | 871,151.672 | 0.40% | 871,452.706 | 0.36% | 844,198.067 |

### 3.3 结果分析
1. 从轻微扰动场景看，各方法 MAPE 均较低（0.09%~0.10%），说明在数据质量较高时四种方法均可用；其中中位数和截尾平均略优于算术平均。
2. 在单点异常场景与缺失数据场景中，鲁棒型方法优势显著：中位数/截尾平均的 MAPE 约为 0.36%~0.40%，明显低于算术平均/加权平均（5.53%~10.61%）。
3. 从链上开销看，算术平均成本最低；中位数与截尾平均相较算术平均约增加 2%~3% 左右，属于可接受的额外开销；加权平均开销最高。
4. 因此，本文方案在三类数据质量条件下均可稳定运行；若场景存在异常值或缺失值，优先采用中位数或截尾平均；若数据较干净且成本敏感，可采用算术平均。

## 4. 实验二：计算任务链上/链下成本对比

### 4.1 实验设置
- 计算实验采用 4 个任务类型：简单运算、浮点运算、随机运算、复杂运算，每类选取 1 个代表任务（含明确 `js_expression`）。
- 场景覆盖工业 KPI 核算、非线性连续公式、随机抽样估计与多约束优化计算。
- 任务来源口径：本实验中的计算任务为“研究性设计任务”，即基于公开数据集字段构造工业计算逻辑，不是数据集原生给出的固定标签任务。
- 对比路径：
  - 直接在链上执行任务（链上计算）
  - 链下完成整条原子任务链后，一次回传链上确认（链下计算）
- 计算表达：每个任务的可执行 JS 表达式写入 `datasets/compute_tasks_10_scenarios.json` 的 `js_expression` 字段。
- 链下确认模式：采用低开销 `lite_single_submit` 模式（单白名单节点一次提交场景结果），用于评估工程可用性与成本收益。

### 4.2 计算场景与任务表达式一览
为避免“仅报开销，不知算了什么”的问题，4 类计算任务的核心表达如下。

| 场景类型 | 代表计算任务 | JS 表达式（核心） |
| --- | --- | --- |
| 简单运算 | 线性结算公式 | `amount = a * b + c` |
| 浮点运算 | 连续非线性评分 | `score = α*log(x+1) + β*exp(y) + γ*sqrt(z)` |
| 随机运算 | 蒙特卡洛概率估计 | `for i in N: if(random() < p) hit++; p_hat = hit / N` |
| 复杂运算 | 方案组合优化 | `result=\min\{(60x+25y+15z-target)^2+2\cdot(400x+120y+80z-budget)^2\}` |

### 4.3 计算任务开销结果
计算任务不同执行路径下的开销对比如下表所示。

| 场景类型 | 代表计算任务 | 复杂性说明 | 链上计算Cost(gas) | 链下计算Cost(gas) | 优化率 |
| --- | --- | --- | ---: | ---: | ---: |
| 简单运算 | 线性结算公式：`amount = a * b + c` | 常数复杂度，约 `1` 次乘法 + `1` 次加法 | 46,183 | 72,033 | -55.97% |
| 浮点运算 | 连续非线性评分：`score = α·log(x+1)+β·exp(y)+γ·sqrt(z)` | 包含 `log/exp/sqrt` 连续函数，需浮点支持 | N/A（链上不支持） | 54,933 | N/A |
| 随机运算 | 蒙特卡洛概率估计：`p ≈ hits / N` | `N=4000` 轮随机抽样、比较与计数 | N/A（链上不支持） | 54,933 | N/A |
| 复杂运算 | 方案组合优化：`\min\{(60x+25y+15z-target)^2+2\cdot(400x+120y+80z-budget)^2\}` | 离散搜索优化（`11×11×11=1331` 组候选组合） | 2,263,134 | 54,933 | 97.57% |

### 4.4 结果分析
1. 简单运算场景中，链下开销高于链上（`-55.97\%`），说明链下提交与确认存在相当固定成本。
2. 浮点运算与随机运算场景在当前 EVM 口径下不适合直接链上实现（表中为 `N/A`），链下执行是必要路径。
3. 复杂运算场景中（组合优化目标函数 + `1331` 组离散搜索），链下优化率达到 `97.57\%`，说明复杂度足够高时链下路径具有显著优势。
4. 因此可得结论：链下计算存在相当的额外开销，但适用于浮点、随机及复杂运算场景。

## 5. 实验三：白名单 + 签名机制安全性验证

### 5.1 实验设置
- 注册预言机节点：5
- 每任务授权节点：3
- 阈值：3
- 轮次：compute 12轮，data 12轮
- 攻击向量：越权提交、错误签名、重放提交（同任务重复提交）

### 5.2 结果汇总
| 指标 | Compute | Data | Overall |
| --- | ---: | ---: | ---: |
| 攻击尝试次数 | 36 | 36 | 72 |
| 拦截次数 | 36 | 36 | 72 |
| 拦截率 | 100% | 100% | 100% |
| 任务完成率 | 100% | 100% | - |
| 结果完整性 | 100% | 100% | - |
| 合法流程平均Gas | 465,715.08 | 729,142.08 | - |

### 5.3 结果分析
1. 三类攻击向量（越权提交、错误签名、重放提交）在实验中均被阻断，表明“白名单 + 结果签名绑定 + 任务状态约束”组合有效。
2. 在高拦截率下，任务完成率与结果完整性仍保持 100%，说明机制未显著牺牲可用性。
3. 该实验验证了应用层防护有效性，但不等价于底层共识或跨链协议级安全证明。

## 6. 综合结论

1. 数据型任务：在公开工业数据驱动的仿真实验中，聚合链路可稳定达成可用性目标；其中中位数方法（MEDIAN）在复杂数据质量场景下表现最稳健。
2. 计算型任务：简单任务宜链上，复杂或链上不易表达任务宜下链；“按复杂度与可表达性选路”可显著降低总体成本并保证可执行性。
3. 安全机制：白名单与签名绑定可有效阻断越权、篡改与重放攻击，且不破坏任务可用性。
4. 工程启示：Oracle 体系应采用“数据聚合策略 + 任务复杂度路由 + 安全校验”三位一体设计。

---

## 7. 有效性威胁与局限

1. 当前主要为 PoC 级受控实验，尚未进行大规模长时稳定性压测。
2. 计算任务复杂度虽覆盖多类工业场景，但仍可进一步扩大任务库与真实在线数据流。
3. 安全实验聚焦应用层攻击拦截，尚未覆盖更复杂的跨系统联动攻击链。
4. 后续可补充多节点并发、故障注入、长期运行漂移等实验以增强外部有效性。
