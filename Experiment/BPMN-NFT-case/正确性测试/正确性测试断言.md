# 正确性测试断言

## 一、断言集合

### 1.操作前置条件正确性断言

#### 1.1**资产存在性断言(分类)**

- **转让型NFT**: transfer / burn / query操作前,token必须存在或已被mint
- **转让型FT**: transfer / burn操作前,tokenName必须已存在(至少被mint过一次)
- **分发型NFT**: transfer / burn / grant usage rights / revoke usage rights / query操作前,token必须存在
- **增值型NFT**: branch / merge / query操作前,引用的源token(refTokenIds)必须存在

#### 1.2**唯一性断言(NFT类资产)**

- **转让型NFT**: 同一tokenId在整个BPMN流程中只能被mint一次(除非已被burn)
- **分发型NFT**: 同一tokenId在整个BPMN流程中只能被mint一次(除非已被burn)
- **增值型NFT**: 同一tokenId在整个BPMN流程中只能被branch或merge创建一次(除非已被burn)
- **不适用于FT**: FT通过tokenName标识,可以多次mint增加供应量

#### 1.3**所有权前置断言(基于业务逻辑)**

- **建模阶段验证**:
  - tokenId在所有DataObjectReference中必须唯一(避免同一流程中多个资产使用相同ID)
  - 同名FT(tokenName相同)应该对应同一种代币,不能有定义冲突

- **实例创建阶段验证**:
  - **转让型NFT**: burn / transfer操作的caller必须是当前owner(基于流程执行顺序推断)
  - **分发型NFT**: burn / grant usage rights / revoke usage rights操作的caller必须是当前owner
  - **增值型NFT**: burn操作的caller必须是当前owner
  - **FT**: burn / transfer操作的caller必须有足够余额(通过模拟执行计算余额)
  - **预存在ERC的token**: 若标记tokenHasExistInERC=true,需通过区块链查询确定初始owner,并验证该owner在参与方绑定中存在

- **参与方绑定验证(基于协作业务需求)**:
  - 所有Task的caller必须在"Binding Participants"中绑定到真实的组织或用户
  - equal类型绑定: 绑定到具体用户,适用于需要明确责任人的操作(如mint、transfer)
  - group类型绑定: 绑定到组织角色,适用于组织内任意成员都可执行的操作
  - 参与方的区块链身份(x509证书)必须与绑定的用户身份匹配(防止身份冒用)

#### 1.4**授权状态前置断言(分发型)**

revoke usage rights前必须对应的callee已被grant usage rights授权

#### 1.5**派生前置断言(增值型)**

- branch和merge操作中,refTokenIds中的所有tokenId必须已存在
- refTokenIds不能为空(merge操作)
- refTokenIds中不能包含自身tokenId(避免循环引用)
- 被引用的token执行派生操作时,不影响源token的所有权

#### 1.6**身份操作权限断言(细粒度)**

- **BPMN参与方层面**: Task的caller必须是BPMN Choreography中定义的Participant
- **组织身份层面**: 参与方在创建实例时必须绑定到联盟中的组织(通过MSP标识)
- **用户身份层面**:
  - equal类型绑定:绑定到具体用户,需要x509证书或Fabric Identity
  - group类型绑定:绑定到组织角色,通过MSP+attributes验证
- **资产操作权限**: Task中的caller参与方必须对DataObjectReference关联的资产有相应操作权限(所有权/余额)
- **跨合约调用权限**: TokenElement操作需要绑定到对应的ERC合约(通过taskERCMap)

### 2.参数完整性与合法性断言

#### 2.1**参数完备性断言(分类)**

- **转让型NFT必须参数**:
  - mint操作: tokenId + tokenName + caller
  - transfer操作: tokenId + tokenName + caller + callee
  - burn操作: tokenId + tokenName + caller
  - query操作: tokenId + tokenName + caller

- **转让型FT必须参数**:
  - mint操作: tokenName + tokenNumber + caller
  - transfer操作: tokenName + tokenNumber + caller + callee
  - burn操作: tokenName + tokenNumber + caller
  - query操作: tokenName + caller

- **分发型NFT必须参数**:
  - mint操作: tokenId + tokenName + caller
  - grant usage rights操作: tokenId + tokenName + caller + callee
  - revoke usage rights操作: tokenId + tokenName + caller + callee
  - transfer操作: tokenId + tokenName + caller + callee
  - burn操作: tokenId + tokenName + caller
  - query操作: tokenId + tokenName + caller

- **增值型NFT必须参数**:
  - branch操作: tokenId + tokenName + caller + refTokenIds
  - merge操作: tokenId + tokenName + caller + refTokenIds
  - burn操作: tokenId + tokenName + caller
  - query操作: tokenId + tokenName + caller

#### 2.2**参数类型断言**

- **资产类型互斥**: DataObjectReference中,assetType只能是以下之一: distributive / transferable / value-added
- **Token类型互斥**: transferable类型下,tokenType只能是NFT或FT
- **参数组合互斥**: 同一DataObjectReference不能同时存在tokenId和tokenNumber
- **NFT与FT区分**:
  - 存在tokenId则为NFT(必须有tokenId和tokenName)
  - 不存在tokenId但有tokenNumber则为FT(必须有tokenName和tokenNumber)

#### 2.3**callee /caller合法性断言**

- **Caller验证**:
  - caller不能为空
  - caller必须是BPMN中定义的Participant ID
  - caller在实例创建时必须被绑定(showBindingParticipantValueMap中存在)

- **Callee验证**:
  - transfer / grant usage rights / revoke usage rights操作时,callee不能为空
  - callee不能等于caller
  - callee中的每个参与方必须是BPMN中定义的Participant ID
  - callee中的每个参与方在实例创建时必须被绑定
  - callee可以是数组(支持多个接收方)

#### 2.4 **refTokens非空断言(增值型)**

- merge操作时refTokenIds不能为空
- branch操作时refTokenIds可以为空或包含源token

#### 2.5**refTokens不自引用断言(增值型)**

refTokenIds中不能含有本身tokenId,即不能自引用

#### 2.6**tokenNumber正值断言(FT)**

FT类型的tokenNumber必须为正整数

#### 2.7**DataObject与Task关联断言**

- **业务语义**: DataObjectReference定义资产,Task定义对资产的操作,二者必须通过数据关联连接
- **约束内容**:
  - Task必须通过DataInputAssociation或DataOutputAssociation连接到DataObjectReference
  - Task的资产信息(tokenName/tokenId/assetType/tokenType等)从连接的DataObjectReference读取
  - translator.py在生成链码时,会合并DataObject和Task的字段(见_merge_task_with_dataobject函数,356-421行)
  - 如果Task未连接DataObject,则向后兼容从Task自身的documentation读取(但不推荐)
- **验证位置**: AssetTaskModal.tsx在打开时检查是否连接DataObject(454行)

### 3.资产状态迁移正确性断言

#### 3.1**生命周期顺序断言(分类)**

- **转让型NFT/FT**:
  ```
  NonExist → Minted → (Transferred)* → (Burned)? → (Re-minted)?
  ```

- **分发型NFT**:
  ```
  NonExist → Minted → [(Granted)* → (Revoked)*]* → (Transferred)* → (Burned)?
  ```

- **增值型NFT**:
  ```
  NonExist → (Branch|Merge with refTokenIds) → (Transferred)* → (Burned)? → (Re-branch|Re-merge)?
  ```

#### 3.2**Burn终止性断言(NFT)**

- **业务语义**: NFT代表唯一资产,一旦burn(销毁)表示该资产不再存在
- **约束内容**:
  - NFT burn后,无法再进行transfer / grant usage rights / revoke usage rights / branch / merge操作
  - 但可以使用相同tokenId重新mint(相当于新资产,新的生命周期开始)
- **FT例外**: FT代表可分割资源,burn只是减少供应量,不影响后续mint

#### 3.3**Transfer单点所有权断言(NFT)**

- **业务语义**: NFT代表不可分割的唯一资产,同一时刻只能有一个所有者
- **约束内容**:
  - 任一时刻,NFT的owner有且只有一个(不能共同拥有)
  - transfer操作必须完整转移所有权: caller失去所有权,callee获得所有权
  - 在BPMN流程中需跟踪每个NFT当前的owner是谁(基于执行顺序推导)
- **FT对比**: FT可以多人持有不同数量,不存在"单点所有权"

#### 3.4**Query不变性断言**

- **业务语义**: 查询(query)是只读操作,不应修改任何状态
- **约束内容**:
  - query操作不改变token的owner
  - query操作不改变任何参与方的余额
  - query操作不改变token的生命周期状态(未burn的仍未burn)
  - query操作可以被任何有权限的参与方执行,无需所有权验证
- **区块链原理**: 查询对应链码的query方法,不产生交易,不修改世界状态

#### 3.5**状态机一致性断言(链码运行时)**

- **业务语义**: BPMN流程执行通过元素状态流转控制,确保顺序正确和不重复执行
- **元素状态定义**(smartcontract.go:59-66):
  ```go
  const (
    DISABLED = 0  // 未激活,不能执行
    ENABLED = 1   // 已激活,可以执行
    WAITINGFORCONFIRMATION = 2  // 等待确认(BusinessRule特有)
    COMPLETED = 3 // 已完成
  )
  ```
- **约束内容**:
  - **TokenElement状态流转**: DISABLED → ENABLED → COMPLETED
    - DISABLED: TokenElement创建时初始状态
    - ENABLED: 前置条件满足(控制流到达,数据依赖满足)时变为ENABLED
    - COMPLETED: ERC操作成功执行后变为COMPLETED
  - **Message状态**: 同步于ChoreographyTask执行
  - **Gateway状态**: 网关激活和完成的状态
  - **BusinessRule状态**: 可能有WAITINGFORCONFIRMATION中间状态
- **状态不可回退**: 状态只能单向前进,不能COMPLETED回到ENABLED(保证操作不重复执行)

### 4.所有权/使用权一致性断言(分发型)

#### 4.1**所有权–使用权分离断言**

- grant usage rights不得改变owner
- 只有owner才能执行grant usage rights和revoke usage rights
- 使用权可以授予多个参与方,但所有权保持单一

#### 4.2**使用权单独可撤销断言**

revoke usage rights仅影响usage,不影响ownership

#### 4.3**使用权生命周期依附断言**

token被burn后,所有使用权全部消失

### 5.数量与守恒正确性断言(FT)

#### 5.1**数量正值断言**

tokenNumber必须为正整数(FT的mint / transfer / burn操作)

#### 5.2**数量充足断言**

- **业务常识**: 不能转移或销毁超过自己持有的数量
- **约束内容**:
  - transfer操作: caller的余额必须 >= tokenNumber (不能透支)
  - burn操作: caller的余额必须 >= tokenNumber
  - 需要在创建实例阶段模拟执行,计算每个参与方在每个Task执行时的余额
- **验证方法**: 按BPMN执行顺序追踪每个参与方的余额变化

#### 5.3**账户隔离断言**

- **业务常识**: FT转账是零和游戏,一方减少的量等于另一方增加的量
- **约束内容**:
  - transfer操作中,caller余额减少tokenNumber
  - transfer操作中,callee余额增加tokenNumber
  - 两者变化量相等且相反,系统总量守恒
  - 不同参与方的账户相互独立,A的余额变化不影响B(除非有转账)
- **物理意义**: 类似银行转账,总金额不变,只是在账户间流转

#### 5.4**余额非负断言**

任何时刻,任何参与方的FT余额必须 >= 0

### 6.资产演化与引用关系正确性(增值型)

#### 6.1**引用不可变断言**

token一旦通过branch或merge建立,其refTokenIds引用关系不可变
- refTokenIds在DataObjectReference中定义(AssetModal.tsx的176-178行)
- 创建后引用关系固化,不可修改

#### 6.2**派生不破坏源断言**

任何被引用的token(在refTokenIds中),其owner不会受到派生操作影响
- branch/merge操作只创建新token
- 源token的ownership保持不变

#### 6.3**引用存在性断言**

refTokenIds中的所有tokenId必须在当前instance中存在(已被mint/branch/merge)

#### 6.4**引用类型一致性断言**

refTokenIds引用的token应该与当前token类型兼容(都是增值型)

## 二、断言的三阶段分类

本章将第一章的所有断言按照验证阶段分类:**建模阶段、创建实例阶段、运行时阶段**。

### 1. 建模阶段断言

**定义**: 在BPMN建模时,通过前端UI约束即可验证的断言,无需运行时信息。

#### 1.1 结构完整性断言
- **1.2 唯一性断言**: tokenId在所有DataObjectReference中不重复
- **2.1 参数完备性断言**: 根据operation类型强制填写必需参数(caller/callee/tokenId/tokenName等)
- **2.2 参数类型断言**: assetType/tokenType互斥,参数组合互斥
- **2.3 caller/callee合法性断言**: caller/callee必须是已定义的Participant
- **2.6 tokenNumber正值断言**: FT的tokenNumber必须为正整数
- **2.7 DataObject与Task关联断言**: Task必须通过DataInputAssociation或DataOutputAssociation连接到DataObjectReference

#### 1.2 引用合法性断言
- **2.4 refTokens非空断言**: merge操作时refTokenIds不能为空
- **2.5 refTokens不自引用断言**: refTokenIds中不能包含自身tokenId
- **6.1 引用不可变断言**: refTokenIds在DataObjectReference中定义后不可修改

**建模阶段特点**: 主要验证**结构正确性**(参数完整、类型正确、引用合法),通过UI约束(下拉框、必填校验、实时检查)实现。

---

### 2. 创建实例阶段断言

**定义**: 在绑定参与方后、部署链码前,通过模拟执行BPMN流程可以验证的断言,需要参与方绑定信息和初始状态。

#### 2.1 前置条件断言
- **1.1 资产存在性断言**: 操作前token必须存在(通过模拟执行追踪)
- **1.2 唯一性断言**: 同一tokenId不能mint两次(除非已burn)
- **1.3 所有权前置断言**: burn/transfer/grant等操作的caller必须是当前owner
- **1.4 授权状态前置断言**: revoke前必须有对应的grant
- **1.5 派生前置断言**: branch/merge的refTokenIds必须已存在
- **1.6 身份操作权限断言**: caller/callee必须在Binding Participants中绑定

#### 2.2 生命周期断言
- **3.1 生命周期顺序断言**: 状态迁移符合定义的生命周期
- **3.2 Burn终止性断言**: burn后不能再有操作(除query和re-mint)
- **3.3 Transfer单点所有权断言**: NFT任一时刻只有一个owner

#### 2.3 数量与余额断言(FT)
- **5.1 数量正值断言**: tokenNumber为正整数
- **5.2 数量充足断言**: transfer/burn时caller余额充足
- **5.3 账户隔离断言**: 转账零和游戏,总量守恒
- **5.4 余额非负断言**: 任何时刻余额>=0

#### 2.4 引用关系断言(增值型)
- **6.2 派生不破坏源断言**: branch/merge不影响源token的owner
- **6.3 引用存在性断言**: refTokenIds中所有token必须存在
- **6.4 引用类型一致性断言**: refTokenIds引用的token类型兼容

**创建实例阶段特点**: 主要验证**运行时状态正确性**(所有权、余额、授权状态),通过**模拟执行**追踪每个Task的前置条件和后置状态。

---

### 3. 运行时阶段断言

**定义**: 在链码执行时,由区块链和ERC合约保证的断言,涉及实际的链上操作和事务。

#### 3.1 链上资产操作断言
- **1.1 资产存在性断言**: ERC合约查询验证token是否存在
- **1.3 所有权前置断言**: ERC合约验证caller是否为owner
- **1.4 授权状态断言**: 分发型usage rights验证

#### 3.2 状态机断言
- **3.5 状态机一致性断言**: TokenElement状态流转DISABLED→ENABLED→COMPLETED,不可回退

#### 3.3 操作语义断言
- **3.4 Query不变性断言**: query操作不修改世界状态
- **4.1 所有权使用权分离断言**: grant不改变owner
- **4.2 使用权单独可撤销断言**: revoke仅影响usage
- **4.3 使用权生命周期依附断言**: burn清除所有usage rights

#### 3.4 数量守恒断言(FT)
- **5.1 数量正值断言**: 链码验证tokenNumber>0
- **5.2 数量充足断言**: ERC合约检查balance
- **5.3 账户隔离断言**: ERC合约实现转账逻辑
- **5.4 余额非负断言**: ERC合约保证balance>=0

#### 3.5 引用存储断言(增值型)
- **6.1 引用不可变断言**: refTokenIds持久化后不可修改
- **6.2 派生不破坏源断言**: branch/merge链码实现保证

**运行时阶段特点**: 主要由**链码逻辑和ERC合约**保证,涉及实际的链上操作、事务回滚、状态机控制

#### 3.0.0 两组织测试架构说明

本测试路径采用**两组织多用户**架构，相比三组织架构具有以下优势：

**架构设计**:
- **OrgA**: 3个用户 (user_A1, user_A2, user_A3)

3. **多方参与场景**: 通过同组织多用户模拟多方协作（如T29多用户授权、T40多层merge）

**断言覆盖保证**:
- 所有原有断言均得到完整覆盖
- 两组织架构足以验证所有权、授权、转移、引用等核心业务逻辑
