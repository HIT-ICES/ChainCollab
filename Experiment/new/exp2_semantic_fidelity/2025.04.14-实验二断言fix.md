# 实验二断言修订说明

## 1. 文档定位

本文档用于说明 `exp2_semantic_fidelity` 中“DSL 到 Go / Solidity 静态一致性验证”的断言设计。

这里的断言来源不再是：

- `src/newTranslator/MDAcheck/check.ocl`

而是直接来自 `newTranslator` 的真实代码生成逻辑：

- `src/newTranslator/CodeGenerator/b2cdsl-go/...`
- `src/newTranslator/CodeGenerator/b2cdsl-solidity/...`

也就是说，本实验二关注的是：

1. **结构一致性**
   - DSL 元素在 Go / Solidity 中是否存在对应结构
2. **控制语义一致性**
   - DSL flow 的 `trigger / branch / action` 是否在 Go / Solidity 中存在对应控制逻辑


## 2. 验证边界

当前实验二只验证**静态一致性**，不直接验证以下内容：

- 不验证运行时行为轨迹一致性
- 不验证 Go 与 Solidity 的执行结果等价
- 不验证外部 DMN / Oracle 返回值的真实业务语义
- 不验证链上部署与执行环境差异

因此，本文中的“保留”“映射”“一致性”均指：

- 可以在目标代码的结构或控制逻辑中找到对应静态证据

而不是：

- 已经证明运行时行为完全等价


## 3. 当前追踪方式

当前实验二采用轻量级静态追踪方式，而不是 generator 侧显式 trace 文件：

- **name-based trace**
  - DSL 元素名映射到代码中的状态字段、枚举项、函数名、处理器名
- **trigger-based trace**
  - DSL `flow.trigger` 映射到 Go / Solidity 中对应 handler 的触发类型

因此，这里的断言适合作为：

- DSL -> 代码的结构/控制语义静态一致性验证

但不应被表述为：

- 强语义追踪
- 运行时等价性证明


## 4. 断言表达方式

下面将断言拆为两张表：

- `A. 结构一致性`
- `B. 控制语义一致性`

每条断言都明确以下信息：

- `ID`
- `断言说明`
- `目标语言`
- `静态证据`
- `通过标准`
- `生成逻辑来源`

## 5. 使用建议

如果实验二当前只宣称“结构一致性 + 控制语义一致性”，那么结论建议表述为：

- DSL 元素在 Go / Solidity 中是否具有对应静态结构
- DSL flow 的 trigger、branch、action 是否在 Go / Solidity 中具有对应控制逻辑证据

不建议直接表述为：

- 已验证 DSL 到两个代码的完整语义等价
- 已验证 Go 与 Solidity 的运行时一致性

后两者需要额外的行为轨迹实验单独支撑。
