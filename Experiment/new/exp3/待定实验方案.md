# 待定实验方案

## 1. 当前想验证的问题

实验三当前聚焦的是**运行时行为一致性**，核心问题不是代码结构像不像，而是：

- 同一份 DSL 派生出的多平台实现
- 在相同输入路径下
- 是否会表现出一致的流程推进行为

这里计划比较三条执行线：

- DSL 参考语义执行线
- Fabric / Go 实际执行线
- Ethereum / Solidity 实际执行线

最终希望回答三个问题：

- `DSL vs Fabric` 是否一致
- `DSL vs Solidity` 是否一致
- `Fabric vs Solidity` 是否一致


## 2. 当前实验主线

目前确认下来的实验主线可以概括为：

**先人为设计一条输入路径，再分别送入 DSL 模拟器、Fabric 环境和 Solidity 环境执行，最后对三方执行轨迹做一致性比较。**

也就是说，实验三不是让 DSL 自己先自动生成路径，而是：

1. 先准备路径输入
2. 用 DSL 参考语义回放这条路径
3. 在两个真实执行环境中回放同一路径
4. 比较三方执行结果


## 3. 输入路径的定位

这里的“输入路径”本质上是**测试用例**，需要人为设计，或者后续由专门的路径生成器生成，但它本身不应由 DSL 模拟器临时决定。

这样设计的原因是要保证三条执行线的输入完全一致，避免比较时混入额外变量。

因此，路径输入在实验三里的角色是：

- 作为三条执行线共享的统一输入
- 明确规定每一步要触发的节点或动作
- 必要时附带该步的输入参数、调用身份或业务数据

DSL 模拟器在这里的职责不是“找路径”，而是：

- 检查给定路径在 DSL 语义下是否可执行
- 产出该路径对应的参考执行轨迹


## 4. 为什么不是让 DSL 自动生成路径

如果让 DSL 模拟器自己生成路径，再拿这个结果去和平台执行做对比，会出现两个问题：

- 比较对象不稳定，因为不同生成策略可能给出不同路径
- 实验重点会从“跨平台一致性”偏成“DSL 自身路径探索”

而实验三真正想做的是控制变量：

- 输入路径固定
- 三条线都执行同一路径
- 比较谁接受、谁拒绝、谁的状态推进不同

所以当前更合理的思路是：

- 路径由研究者手工设计为主
- DSL、Fabric、Solidity 都只负责执行和记录

后续当然可以扩展出“候选路径自动生成器”，但它的身份应是**测试用例生成工具**，而不是实验三的基线语义本身。


## 5. 三条执行线分别做什么

### 5.1 DSL 线

DSL 线的作用是提供参考语义。

它负责：

- 读取 DSL 模型
- 按给定路径逐步回放
- 在每一步判断当前动作是否可触发
- 记录执行前后的状态变化
- 输出统一格式的参考轨迹

这条线的结果相当于“标准答案”。


### 5.2 Fabric / Go 线

Fabric 线的作用是提供 Go 代码在真实平台上的运行结果。

它负责：

- 接收与 DSL 相同的路径输入
- 在 Fabric 环境中按顺序执行
- 记录每一步的真实执行反馈
- 输出可与 DSL 对齐的执行轨迹


### 5.3 Ethereum / Solidity 线

Solidity 线的作用是提供 Solidity 代码在真实平台上的运行结果。

它负责：

- 接收与 DSL 相同的路径输入
- 在 Ethereum / FireFly 环境中按顺序执行
- 记录每一步的真实执行反馈
- 输出可与 DSL 对齐的执行轨迹


## 6. 三方真正要比什么

实验三不是只比较“最后有没有结束”，而是比较**整条执行轨迹**。

建议至少比较以下内容：

- 每一步是否被接受或拒绝
- 每一步执行前的可触发节点集合
- 条件判断结果是否一致
- 节点状态变化是否一致
- 全局变量变化是否一致
- 最终状态是否一致
- 若拒绝，拒绝原因是否一致

因此，实验三的比较对象更准确地说是：

- `参考轨迹`
- `Fabric 实际轨迹`
- `Solidity 实际轨迹`

而不是只比一个最终结果。


## 7. 当前可以怎么组织实验

当前建议按下面的结构来组织：

### 第一步：设计代表性路径

优先手工设计一批有代表性的路径，例如：

- 正常顺序执行路径
- 含排他网关分支的路径
- 含并行分叉 / 汇聚的路径
- 含 business rule 的路径
- 含 oracle task 的路径
- 应被拒绝的非法路径


### 第二步：先跑 DSL 参考执行

用 DSL 模拟器对这些路径逐条回放，得到参考轨迹，并确认：

- 该路径在 DSL 语义下是否应被接受
- 每一步状态如何推进


### 第三步：再跑两个真实环境

把同样的路径输入送到：

- Fabric / Go 环境
- Ethereum / Solidity 环境

分别记录实际执行轨迹。


### 第四步：做三方对比

围绕每条路径输出比较结果，例如：

- 三方全部一致
- 平台接受但 DSL 应拒绝
- 最终结果一致但中间状态推进不同
- 分支选择一致但变量变化不同


## 8. 当前实现状态

需要特别说明的是，**当前文档里的目标方案和代码里的实际完成度并不完全一样**。

就目前情况看：

- DSL 参考模拟执行这条线已经比较明确
- Solidity / FireFly 这条线已有单独回放脚本基础
- Fabric 线还更多处于预留和待补阶段
- 三方统一比较器还没有完全串成一个最终入口

所以现阶段更准确的说法应是：

> 实验三的目标方案已经明确为“共享输入路径 + 三方轨迹对比”，其中 DSL 线最完整，Solidity 线已有基础，Fabric 线和统一比较部分仍待继续实现。


## 9. 一句话介绍版本

如果需要对外简单介绍，目前可以这样说：

> 实验三准备采用“共享路径输入、三方轨迹对比”的方法：先人工设计代表性执行路径，再分别在 DSL 参考语义、Fabric/Go 实现和 Solidity/Ethereum 实现中回放同一路径，最后比较三方在步骤接受性、状态推进、变量变化和最终结果上的一致性。


## 10. 后续可扩展方向

后续可以逐步补充：

- 路径自动生成器，用于批量构造合法 / 非法测试路径
- Fabric 执行脚本与轨迹输出
- 三方统一轨迹格式
- 自动差异分析与报告生成
- 更多边界情况，例如循环、异常回滚、复杂条件分支

但这些都建立在当前主线不变的前提上：

**先固定输入路径，再比较 DSL、Fabric、Solidity 的执行轨迹是否一致。**
