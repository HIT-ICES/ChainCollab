# Exp3: 基于 DSL 参考语义的状态空间遍历方案

## 1. 目标

本方案用于把 `exp3` 从“人工挑路径再对比”升级为：

- 先从 DSL 定义一套**参考语义**
- 再基于参考语义做**状态空间遍历**
- 自动生成路径与参考轨迹
- 最后把同一批路径回放到 `DSL / Fabric / Solidity` 三条执行线

这样，实验结论可以从“代表性路径对比”提升为：

- 对单个 DSL 模型，给出**有界完备**的路径覆盖声明
- 对 case 集合，给出**结构化覆盖**声明


## 2. 先说清楚覆盖声明

这里不直接宣称“覆盖所有 DSL 表达的情况”，而是区分两层：

### 2.1 单个 DSL 模型上的覆盖

对某个具体模型 `M`，若其运行状态空间有限，则可以穷举所有可达路径；  
若存在循环、无界变量或外部输入无界，则只声明：

> 覆盖所有长度不超过 `k` 的可达执行路径。

这叫**有界完备**。

### 2.2 DSL 语言层面的覆盖

对整个 DSL 语言，通常不证明“绝对全覆盖”，而是声明：

- 选择一组覆盖主要语义构造的基准模型
- 对每个模型做有界完备遍历

因此，`exp3` 更合适的表述是：

> 对基准 DSL case 集，在既定边界下覆盖每个 case 的全部可达执行情况。


## 3. 从 DSL 到参考语义

参考语义的本质是把 `.b2c` 模型解释成一个**状态迁移系统**。

形式上可写成：

`TS = (S, s0, Act, ->)`

其中：

- `S`：运行状态集合
- `s0`：初始状态
- `Act`：可触发动作集合
- `->`：状态迁移关系


## 4. 静态抽取：从 DSL 解析出什么

先从 `.b2c` 提取以下静态信息。

### 4.1 元素集合

- `Messages`
- `Gateways`
- `Events`
- `BusinessRules`
- `OracleTasks`
- `Globals`

### 4.2 初始信息

- 每个元素的 `initial state`
- 全局变量类型
- `flows` 中的所有触发规则与动作规则

### 4.3 flow 规则分类

根据当前 DSL 语法，主要有这些规则：

- `start event E enables X`
- `when message M sent/completed then ...`
- `when gateway G completed then ...`
- `when gateway G completed choose { ... }`
- `when businessrule R ready/waiting/done then ...`
- `when oracletask O done then ...`
- `when event E completed then ...`
- `parallel gateway G await A, B, ... then ...`

动作集合为：

- `enable X`
- `disable X`
- `set Global = Literal`


## 5. 运行状态怎么定义

建议把运行状态定义成下面这个元组：

`State = (elem_state, globals, join_mem, trace_meta)`

其中：

### 5.1 `elem_state`

记录每个流程元素当前状态：

- `INACTIVE`
- `READY`
- `PENDING_CONFIRMATION`
- `DONE`

这是最核心的状态部分。

### 5.2 `globals`

记录所有全局变量当前值。

例如：

```json
{
  "Counter": 1,
  "Retry": false
}
```

### 5.3 `join_mem`

用于支持并行汇聚。

虽然很多场景下可以直接通过元素状态判断 `await` 源是否都已完成，但为了避免：

- 循环后重复进入同一个 join
- 多次激活带来的歧义

建议显式记录：

- 某个 `parallel gateway` 当前轮次已到达了哪些源

例如：

```json
{
  "ParallelGateway_2": ["MsgA", "MsgB"]
}
```

### 5.4 `trace_meta`

用于记录非业务状态但与遍历有关的信息，例如：

- 当前路径长度
- 已执行步序号
- 循环展开次数统计

如果实现里想保持状态最小，也可以把这些信息放在遍历器而不是语义状态里。


## 6. 动作字母表怎么定义

为了让 DSL、Fabric、Solidity 三方能对齐，建议把一步执行统一建模成：

`Action = (kind, id, phase, input)`

例如：

- `(event, StartEvent_1, completed, {})`
- `(message, Offer, completed, {"sender":"Seller"})`
- `(gateway, Decide, completed, {})`
- `(businessrule, Rule1, done, {"outputs":{"priceOK":true}})`
- `(oracletask, Oracle1, done, {"outputs":{"oraclePrice":100}})`

这里特别注意：

### 6.1 `message` 既可能有 `sent`，也可能有 `completed`

因为 DSL grammar 允许：

- `when message M sent`
- `when message M completed`

所以参考语义里不能把 message 只看成一个单相动作。

### 6.2 `businessrule` 和 `oracletask` 带外部结果

规则引擎和 oracle 的输出不能凭空出现，建议作为动作输入的一部分显式给出。

否则：

- DSL 模拟器不知道该把哪些 global 改成什么
- 三个平台也无法复现同一输入


## 7. 迁移规则怎么定义

每次迁移都按统一顺序处理：

1. 判断某个动作在当前状态是否可触发
2. 更新被触发元素自身状态
3. 查找所有因该动作满足条件的 flow 规则
4. 顺序执行这些 flow actions
5. 得到后继状态，并记录一步轨迹

下面给出建议语义。

### 7.1 Start Event

若 `StartEvent_x` 为 `READY`，则可触发 `(event, StartEvent_x, completed, {})`

触发后：

- `StartEvent_x: READY -> DONE`
- 执行对应 `start event ... enables ...`

### 7.2 Message

如果模型只关心“完成”而不区分“发送确认”，可先采用简化版：

- `READY` 的 message 可触发 `completed`
- 触发后 `READY -> DONE`

如果后续要贴近链上实现，可扩成两阶段：

- `READY -> PENDING_CONFIRMATION` 对应 `sent`
- `PENDING_CONFIRMATION -> DONE` 对应 `completed`

是否采用两阶段，必须在 `exp3` 里固定下来，三条执行线保持一致。

### 7.3 Gateway

`READY` 的 gateway 可触发 `completed`。

触发后：

- `READY -> DONE`
- 若是普通 `then` 规则，执行动作列表
- 若是 `choose` 规则，先计算 guard，再选择分支

### 7.4 Exclusive `choose`

建议把 `choose` 明确成**确定性语义**：

- 按 DSL 中分支书写顺序检查
- 取第一个满足条件的分支
- 若都不满足且有 `else`，走 `else`
- 若都不满足且无 `else`，该步拒绝

如果一个 case 存在“多个分支都满足”的歧义，建议：

- 直接标为语义歧义 case，不作为 exp3 基线 case

这样可以避免平台实现差异其实来自 DSL 自身未定性。

### 7.5 Parallel Join

`parallel gateway G await A, B, ... then ...` 的建议语义：

- `G` 必须已经 `READY`
- 每当某个 source 完成，就把它记入 `join_mem[G]`
- 当且仅当 `await` 列表全部到齐时，`G` 才可触发 `completed`
- 触发后：
  - `G: READY -> DONE`
  - 执行 `then` 动作
  - 清空本轮 `join_mem[G]`

这样能明确区分：

- “gateway 被 enable 了”
- “gateway 已满足汇聚条件，可以 completed”

### 7.6 BusinessRule

建议把 business rule 执行抽象成两步或一步。

最小实现可先用一步：

- `READY` 的 rule 触发 `(businessrule, R, done, outputs)`
- 根据 `output mapping` 更新 globals
- `READY -> DONE`
- 执行 `when businessrule R done then ...`

若后续要区分排队/执行中，可再引入：

- `READY -> PENDING_CONFIRMATION` 对应 `waiting`
- `PENDING_CONFIRMATION -> DONE` 对应 `done`

### 7.7 OracleTask

与 business rule 类似：

- `READY` 的 task 触发 `(oracletask, O, done, outputs)`
- 应用 `output mapping`
- `READY -> DONE`
- 执行后续 flow actions

### 7.8 Enable / Disable / Set

动作建议这样解释：

- `enable X`
  - 若 `X == INACTIVE`，则变为 `READY`
  - 若 `X` 已是 `READY` 或 `DONE`，是否允许重复 enable 必须固定
  - 建议基线语义里把重复 enable 视为幂等

- `disable X`
  - 若 `X` 尚未完成，则置为 `INACTIVE`
  - 若 `X == DONE`，建议保持 `DONE`

- `set v = literal`
  - 直接覆盖全局变量值


## 8. 如何处理“无界”问题

状态空间遍历能否完备，关键看状态空间是不是有限。

### 8.1 循环

若模型含循环，则需要加边界：

- 最大路径长度 `k`
- 或最大循环展开次数 `r`

### 8.2 全局变量取值域

若 `int/string` 无界，状态空间也无界。建议采用**有限抽象域**：

- `bool`：`{true, false}`
- `int`：取模型中出现的所有整数字面量，再加必要缺省值
- `string`：取模型中出现的字符串字面量，再加必要缺省值

例如 `set Counter = 1`、guard 中有 `Counter == 2`，则可令：

- `Counter` 的抽象域为 `{0,1,2}`

### 8.3 外部结果

对 `businessrule` / `oracletask` 输出，也要先给定有限候选集合。

否则无法完成遍历。


## 9. 状态空间遍历算法

建议先实现 `BFS` 或 `DFS` 的**有界前缀展开**版本。

- 若目标是“先拿到较短路径”，优先 `BFS`
- 若目标是“实现简单、便于递归输出全部路径”，优先 `DFS`

这里要特别注意：

> 如果目标是生成“所有路径”，不能只按运行状态去重；否则不同前缀到达同一状态时，会被错误合并。

因此，遍历器要区分两个层次：

- **状态缓存**：避免重复计算某个状态的可触发动作与后继
- **路径展开**：保留不同前缀，直到达到长度边界 `k`

### 9.1 遍历输入

- 一个 `.b2c` 模型
- 一个变量抽象域定义
- 一个外部输出候选集定义
- 一个长度上界 `k`

### 9.2 遍历输出

- 所有长度 `<= k` 的可达路径
- 每条路径对应的 DSL 参考轨迹
- 可选：每个可达状态的 canonical 编码

### 9.3 伪代码

```text
queue <- [(s0, empty_trace)]
succ_cache <- {}
all_traces <- []

while queue not empty:
    (s, trace) <- pop_front(queue)

    if is_terminal(s) or len(trace) == k:
        all_traces.add(trace)
        continue

    key <- canon(s)
    if key not in succ_cache:
        succ_cache[key] <- []
        for a in enabled_actions(s):
            s2, step_record <- step(s, a)
            succ_cache[key].append((a, s2, step_record))

    if succ_cache[key] is empty:
        all_traces.add(trace)
        continue

    for (_, s2, step_record) in succ_cache[key]:
        trace2 <- trace + [step_record]
        push_back(queue, (s2, trace2))
```

### 9.4 `enabled_actions(s)` 做什么

它要枚举当前状态下所有合法下一步，包括：

- 所有 `READY` 且可完成的 event/message/gateway
- 所有满足输入条件的 businessrule/oracletask 执行动作
- 所有满足 `parallel join` 条件的 gateway 完成动作

如果采用 message 两阶段模型，还要额外枚举：

- `sent`
- `completed`

### 9.5 `canon(s)` 做什么

把状态规范化，作为缓存键，至少包含：

- 所有元素状态
- 所有 globals 当前值
- 所有 `join_mem`

这样遍历器能识别“虽然路径不同，但已经回到同一运行状态”。

但这里的用途应是：

- 复用该状态的后继计算结果
- 统计可达状态数

而不是直接丢弃后续路径前缀。


## 10. 如何声明正确性与覆盖

这个方案至少可以给出两个性质。

### 10.1 Soundness

生成器输出的每条路径，都是 DSL 参考语义下的合法执行路径。

原因是：

- 每一步都只从 `enabled_actions(s)` 中选动作
- 每一步都由 `step(s, a)` 按参考语义推进

### 10.2 Bounded Completeness

在给定：

- 有限抽象域
- 固定外部结果候选集
- 路径长度上界 `k`

的前提下，遍历器会生成所有长度不超过 `k` 的可达路径。

这就是 exp3 里最应该写的覆盖声明。

不要直接写“覆盖所有情况”，而应写：

> 覆盖所有长度不超过 `k` 的参考语义可达路径。


## 11. 非法路径怎么生成

exp3 不只比较合法路径，也要比较“应拒绝路径”。

建议从每个可达状态 `s` 自动构造一批一跳非法动作：

- 选择一个当前不在 `enabled_actions(s)` 中的动作
- 拼到当前合法前缀后面
- 得到一条“前缀合法、最后一步非法”的测试路径

例如：

- 当前 `Accept` 未被 enable，却强行执行 `Accept`
- 并行汇聚未满足时，强行执行 join gateway
- `choose` 还没到 gateway 完成时，强行走分支后的 message

这样生成的非法路径更系统，也更容易说明拒绝测试覆盖面。


## 12. 用现有 case 做基线示例

可以直接复用 `exp2_semantic_fidelity/cases/reference_semantic` 中的 case。

### 12.1 `set_disable_case.b2c`

它体现了：

- `message completed`
- `set global`
- `exclusive choose`
- `disable`

在参考语义下，若 `Retry == true`，应得到主路径：

`StartEvent_1 -> Offer -> Decide -> Cancel -> EndEvent_1`

若 `Retry == false`，应得到：

`StartEvent_1 -> Offer -> Decide -> Accept -> EndEvent_1`

这正好能验证：

- guard 选择
- 分支互斥
- `disable Accept`

### 12.2 `parallel_join_case.b2c`

它体现了：

- parallel split
- source completion reordering
- parallel join

这里至少应覆盖两种关键中间次序：

- `StartEvent_1 -> ParallelGateway_1 -> MsgA -> MsgB -> ParallelGateway_2 -> MsgC -> EndEvent_1`
- `StartEvent_1 -> ParallelGateway_1 -> MsgB -> MsgA -> ParallelGateway_2 -> MsgC -> EndEvent_1`

这样才能证明遍历器没有把并发只当成单一顺序。

## 14. 建议的实现顺序

按下面顺序做，风险最低。

### 第一阶段：先做最小参考语义

先只支持：

- start event
- message completed
- gateway completed
- exclusive choose
- parallel join
- enable / disable / set

暂时不做：

- message `sent`
- businessrule `waiting`
- oracle task 的复杂外部交互
- 循环

### 第二阶段：先跑小型 reference cases

优先跑：

- `set_disable_case.b2c`
- `parallel_join_case.b2c`
- 一个 `event_flow_case`

目标不是马上接链，而是先验证：

- 状态定义对不对
- 遍历结果是否符合直觉
- 轨迹格式是否稳定

### 第三阶段：接入 Go / Solidity 回放

把自动生成的 `legal_paths.json` 与 `illegal_paths.json` 送给：

- DSL 模拟器
- Fabric/Go 执行器
- Solidity 执行器

三方输出统一轨迹后再比对。

### 第四阶段：逐步扩 DSL 语义

后续再补：

- message `sent/completed` 两阶段
- businessrule `ready/waiting/done`
- oracletask 输出映射
- 有界循环展开


## 15. 实验报告里的建议表述

论文或 README 里建议用下面这种说法：

> 我们首先从 B2CDSL 构造参考状态迁移系统，并在有限抽象域与长度边界下对其进行状态空间遍历，生成所有长度不超过 `k` 的可达合法路径，以及由可达前缀扩展得到的代表性非法路径。随后将同一批路径分别回放到 DSL 参考执行器、Fabric/Go 实现和 Solidity 实现中，比较三方在步骤接受性、guard 判断、状态推进、全局变量更新和终态上的一致性。

这个表述比“人工挑几条路径”严谨得多，也不会把结论说过头。


## 16. 最关键的边界条件

最后必须在文档里明确写出以下边界，否则别人会直接质疑“全覆盖”。

- 只保证**有界完备**，不宣称无限路径全覆盖
- 全局变量使用有限抽象域
- 外部 rule/oracle 输出使用有限候选集
- 有歧义的 `choose` case 不纳入基线
- 循环按 `k` 或 `r` 截断

只要这几个边界写清楚，这套方案就足够作为 `exp3` 的正式实验方法。
