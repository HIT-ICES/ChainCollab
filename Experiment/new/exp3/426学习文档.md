# 426 学习文档：Exp3 DSL 与 Solidity 执行轨迹对齐

## 一句话目标

实验三要验证：同一个业务流程、同一条执行路径，在 **DSL 模拟执行器** 和 **Geth/Solidity 自动执行脚本** 中是否表现出一致的流程推进轨迹。

当前已经跑通的是 `SupplyChainPaper`：

- BPMN 输入案例：`/root/code/ChainCollab/Experiment/BPMNwithDMNcase/SupplyChainPaper.bpmn`
- DSL/B2C 语义源：`/root/code/ChainCollab/Experiment/new/exp2_semantic_verification/results/cases/SupplyChainPaper/dsl.b2c`
- Solidity replay 输出：`/root/code/ChainCollab/Experiment/new/exp3/solidity/SupplyChainPaper/paths/paper_very_low_delivery_path/replays/replay_20260426_203413.json`
- DSL normalized 输出：`/root/code/ChainCollab/Experiment/new/exp3/outputs/SupplyChainPaper/paths/paper_very_low_delivery_path/normalized/dsl.normalized.json`
- Solidity normalized 输出：`/root/code/ChainCollab/Experiment/new/exp3/outputs/SupplyChainPaper/paths/paper_very_low_delivery_path/normalized/solidity.normalized.json`

## 总体流程

完整链路是：

```text
执行路径输入
  -> DSL 模拟执行
  -> Solidity/Geth 自动执行
  -> 两边原始轨迹分别归一化
  -> 比较 normalized trace
```

这几个阶段是独立的：

- DSL 模拟执行不会调用链。
- Solidity 自动执行不会调用 DSL 模拟器。
- 归一化脚本只做格式转换，不改变原始执行结果。
- 对比脚本/命令只读取 normalized JSON，不反向影响执行。

## 1. 执行路径输入

现在采用一份平台无关的统一路径源：

```text
/root/code/ChainCollab/Experiment/new/exp3/cases/SupplyChainPaper/paths/paper_very_low_delivery_path/logical_path.json
```

以后优先维护这个文件。它描述的是“业务上这一条路径应该经过哪些元素”，同时允许携带两类执行所需字段：

- `payload`：message / gateway / event / Solidity business rule request 使用。
- `outputs`：DSL 模拟 business rule 输出使用。
- `continue_payload`：Solidity business rule continue 使用。

从这份 logical path 派生两个平台输入的脚本是：

```text
/root/code/ChainCollab/Experiment/new/exp3/scripts/materialize_logical_path.py
```

生成命令：

```bash
cd /root/code/ChainCollab
/root/code/ChainCollab/src/newTranslator/.venv/bin/python \
  /root/code/ChainCollab/Experiment/new/exp3/scripts/materialize_logical_path.py \
  --logical-path /root/code/ChainCollab/Experiment/new/exp3/cases/SupplyChainPaper/paths/paper_very_low_delivery_path/logical_path.json \
  --model /root/code/ChainCollab/Experiment/new/exp2_semantic_verification/results/cases/SupplyChainPaper/dsl.b2c \
  --dsl-case-output /root/code/ChainCollab/Experiment/new/exp3/dsl/SupplyChainPaper/paths/paper_very_low_delivery_path/case.json \
  --solidity-sequence-output /root/code/ChainCollab/Experiment/new/exp3/solidity/SupplyChainPaper/paths/paper_very_low_delivery_path/execution_sequence.json
```

这个命令的作用不是“直接运行实验”，而是把一份平台无关的 `logical_path.json` 派生成两个平台各自可消费的输入文件：

- `case.json`
  - 给 DSL 模拟器使用的路径输入
- `execution_sequence.json`
  - 给 Solidity 自动执行器使用的步骤序列输入

也就是说，这一步生成的是“平台输入”，不是最终轨迹结果、对比结果或报告。

可以把它理解成：

```text
logical_path.json
  -> materialize_logical_path.py
  -> DSL case.json
  -> Solidity execution_sequence.json
```

其中两份输出并不完全相同，因为两个平台的执行口径不同：

- DSL `case.json`
  - 会省略 DSL 模拟器会自动 bootstrap 的 `start event`
  - 会省略 DSL 语义里自动完成的 `parallel join gateway`
- Solidity `execution_sequence.json`
  - 会保留这些步骤，因为链上执行需要显式 invoke

因此，`materialize_logical_path.py` 做的不是简单复制，而是把同一条 logical path 翻译成两个平台各自可执行的输入格式。

生成后，DSL 模拟执行使用：

```text
/root/code/ChainCollab/Experiment/new/exp3/dsl/SupplyChainPaper/paths/paper_very_low_delivery_path/case.json
```

Solidity 自动执行使用：

```text
/root/code/ChainCollab/Experiment/new/exp3/solidity/SupplyChainPaper/paths/paper_very_low_delivery_path/execution_sequence.json
```

也就是说，现在不是手工维护两份路径，而是：

```text
SupplyChainPaper.logical_path.json
  -> materialize_logical_path.py
  -> dsl/SupplyChainPaper/paths/paper_very_low_delivery_path/case.json
  -> solidity/SupplyChainPaper/paths/paper_very_low_delivery_path/execution_sequence.json
```

logical path 中的单步示例：

```json
{
  "type": "message",
  "element": "Message_1wswgqu",
  "payload": {
    "orderDetails": "Bulk order for finished goods"
  }
}
```

当前完整逻辑路径是：

```text

Message_1wswgqu
Message_1ajdm9l
Gateway_0onpe6x
Message_0cba4t6
Message_0pm90nx
Gateway_1fbifca
Message_0rwz1km

Message_0d2xte5

```

派生时有一个重要平台差异：DSL 模拟器会自动 bootstrap start event，并且 parallel join 在 DSL 里是自动完成的。因此 materialize 后的 DSL case 不显式写 `Event_06sexe6` 和 `Gateway_1fbifca`；Solidity execution sequence 会保留它们，因为链上执行需要显式 invoke。归一化时会把 DSL 自动完成的 parallel join 补齐成可对比的 logical step。

如果不想手工写路径，可以用自动路径生成脚本：

```text
/root/code/ChainCollab/Experiment/new/exp3/scripts/generate_dsl_paths.py
```

它的作用是：

- 从 B2C/DSL 模型自动分析“下一步哪些元素可执行”
- 递归展开不同执行分支
- 生成一批平台无关的合法执行路径
- 供后续流程继续派生 `logical_path.json`、DSL `case.json` 和 Solidity `execution_sequence.json`

也就是说，这一步做的不是链上执行，而是**先在 DSL 参考语义层把路径空间展开出来**，作为后续双平台回放的共同输入来源。

它从 B2C/DSL 模型出发，使用 DSL 模拟器语义递归枚举当前 `READY` 元素，自动生成 **bounded executable paths**。这里的 `bounded` 很重要，表示它不是无限意义上的“所有路径证明”，而是在当前边界和抽象下做路径枚举。对 `SupplyChainPaper` 的运行命令：

```bash
cd /root/code/ChainCollab
/root/code/ChainCollab/src/newTranslator/.venv/bin/python \
  /root/code/ChainCollab/Experiment/new/exp3/scripts/generate_dsl_paths.py \
  --model /root/code/ChainCollab/Experiment/new/exp2_semantic_verification/results/cases/SupplyChainPaper/dsl.b2c \
  --case-name SupplyChainPaper \
  --output /root/code/ChainCollab/Experiment/new/exp3/cases/SupplyChainPaper/generated_paths.json \
  --max-depth 64 \
  --max-paths 1000
```

当前 `SupplyChainPaper` 会生成 8 条可执行路径：

- parallel split 后两个 message 的执行顺序有 2 种。
- business rule 的 `FinalPriority` 分支有 4 种：`Low`、`High`、`Medium`、`VeryLow`。
- 因此总数是 `2 * 4 = 8`。

输出文件：

```text
/root/code/ChainCollab/Experiment/new/exp3/cases/SupplyChainPaper/generated_paths.json
```

需要注意，这里的“自动枚举路径”有几个边界：

- 它枚举的是**合法可执行路径**，不是非法/拒绝路径全集。
- 它受 `--max-depth` 和 `--max-paths` 限制，因此更准确的说法是“有界路径枚举”。
- 如果模型里存在循环，当前实现不会无限展开，而是通过状态重复截断和深度上界停止继续搜索。
- `gateway` 的 `compare/else` 分支可参与枚举；`expression` 型分支当前模拟器不求值，因此不会被当作可命中分支。

因此，文档里更严谨的表述应是：

> `generate_dsl_paths.py` 会根据 DSL 模拟器语义，自动枚举当前模型在给定边界下的可执行路径，并把这些路径作为 exp3 后续平台对照执行的统一来源。

## 5. 标准执行格式

normalized trace 顶层格式：

```json
{
  "schema_version": "exp3.unified_trace.v1",
  "platform": "dsl|solidity",
  "case_name": "SupplyChainPaper",
  "path_name": "...",
  "source_trace": "...",
  "element_order": {},
  "raw_steps": [],
  "steps": [],
  "final_state": {}
}
```

每个标准 step 的核心格式：

```json
{
  "index": 16,
  "phase": "execute",
  "trigger": {
    "type": "event",
    "element": "Event_13pbqdz",
    "method": "Event_13pbqdz"
  },
  "enabled_before": [
    "Event_13pbqdz"
  ],
  "accepted": true,
  "rejection_reason": "",
  "payload": {},
  "outputs": {},
  "state_diff": {
    "Event_13pbqdz": [
      "READY",
      "DONE"
    ]
  },
  "global_diff": {},
  "states_after": {},
  "raw": {}
}
```

`states_after` 在真实文件里包含所有元素状态，不只是最后一个元素。

## 6. 两个特殊对齐规则

### 6.1 Business rule 折叠

Solidity 原始执行中：

```text
Activity_0rm8bkp
Activity_0rm8bkp_Continue
```

normalized 后折叠成：

```text
Activity_0rm8bkp
```

原因：DSL 模拟器把 business rule 看作一个逻辑步骤，而链上合约需要 request 和 continue 两笔交易。

### 6.2 Parallel join 补齐

DSL 原始模拟中，`Gateway_1fbifca` 是 parallel join，满足两个前置消息完成后自动完成：

```text
Message_0cba4t6 DONE
Message_0pm90nx DONE
=> Gateway_1fbifca DONE
=> Message_0rwz1km READY
```

Solidity 中需要显式执行：

```text
Gateway_1fbifca
```

所以 normalized DSL trace 里会补一条：

```text
phase = synthetic_parallel_join
trigger.element = Gateway_1fbifca
```

这样两边 logical steps 才能逐步对齐。

## 7. 轨迹对比

现在已经有正式对比脚本：

```text
/root/code/ChainCollab/Experiment/new/exp3/scripts/compare_normalized_traces.py
```

它读取两份 normalized trace，比较：

- logical step 数量
- 每一步触发元素 `trigger`
- 每一步执行前可触发集合 `enabled_before`
- 每一步接受/拒绝结果 `accepted`
- 每一步状态变化 `state_diff`
- 最终状态 `final_state.status`
- 最终所有元素状态 `final_state.element_states`
- 最终剩余可触发集合 `final_state.enabled_elements`


## 10. 读项目时的心智模型

你可以把当前实验看成三层：

第一层是执行：

- DSL 模拟器执行 DSL 参考语义。
- Solidity 脚本执行真实链上合约。

第二层是记录：

- DSL 输出自己的原始 trace。
- Solidity 输出自己的链上 replay JSON。

第三层是对齐：

- normalized trace 把两个不同世界的输出翻译成同一种比较语言。
- 后续对比只看 normalized trace。

最重要的一点：归一化不是执行，不改变事实；它只是把事实重新排版成可比较的结构。

## 11. Chainlink LINK 余额注意事项

链上执行到 business rule / DMN 节点时，会通过 `dmnLiteAddress` 对应的 `MyChainlinkRequesterDMN_Lite` 合约发起 Chainlink 请求。每次请求会消耗 LINK，目前部署里的 fee 是 `0.1 LINK`。

如果 Solidity replay 在业务规则节点报错：

```text
ERC20: transfer amount exceeds balance
```

优先检查 `dmnLiteAddress` 的 LINK 余额。这通常不是 BPMN/DMN 路径参数问题，而是请求合约没有足够 LINK 支付 oracle 请求。

当前环境可以用下面方式补充 LINK：

```bash
cp /root/code/ChainCollab/src/runtime/deployment/deployment.json \
  /root/code/ChainCollab/src/oracle/CHAINLINK/deployment/deployment.json

cd /root/code/ChainCollab/src/oracle/CHAINLINK
RPC_URL=http://localhost:9603 node scripts/fund-contract.js
```

连续跑多条路径前，建议先确认余额足够；否则前几条路径成功后，后面的路径可能因为 LINK 被消耗完而失败。
