# 实验方案总览

本文围绕“**BPMN → DSL → 多平台代码**”的模型驱动链路，设计四组实验，分别从**模型一致性、代码静态语义保真、运行时行为一致性、平台性能差异**四个层面验证方法的有效性。

整体实验链路如下：

**实验1** 验证 BPMN 到 DSL 的一致性；
**实验2** 验证 DSL 到 Go/Fabric 与 Solidity/Geth 的静态语义保真；
**实验3** 验证多平台实现的运行时执行轨迹是否与 DSL 参考语义一致；
**实验4** 比较 Fabric 与 Geth 两类平台的性能差异。

# 实验1：BPMN→DSL 一致性验证

实验1全部聚焦于 **BPMN→DSL** 转换过程，分为 1A、1B、1C 三个层次，分别验证目标模型合法性、源目标结构一致性和源目标行为一致性。

## 1A 目标模型合法性

### 实验目标

验证转换得到的 DSL 模型是否满足良构性约束，是否可作为后续代码生成器的合法输入。

### 主要验证内容

检查 DSL 模型是否满足以下性质：

* 语法结构完整，无缺失块或非法嵌套
* 引用关系有效，无悬空引用
* 元素类型合法，例如 event、gateway、message、businessrule 等使用正确
* flow 结构合法，不存在明显非法连接
* 满足生成器依赖的静态前提

### 实现方法

这一部分直接基于你现有的 `check.ocl` 思路实现即可。
也就是：

* 将转换得到的 DSL 模型转为 XMI/EMF 实例
* 使用 OCL 约束对 DSL 实例进行校验
* 统计每个模型是否通过，失败时记录违反的约束项

### 输出结果

输出以下结果：

* 总模型数
* 通过数 / 未通过数
* 每个模型的 violation 数量
* violation 类型分布

### 实验结论含义

这一部分证明的是：

> 转换后的 DSL 输出模型本身是合法的、良构的，并满足后续代码生成所需的静态约束。

---

## 1B 源目标结构一致性

### 实验目标

验证 BPMN 中的核心结构元素，是否在 DSL 中得到了完整且正确的保留。

### 主要验证内容

重点检查 BPMN 与 DSL 之间的映射契约，包括但不限于：

* Pool / Lane 是否映射为 DSL participant
* Message Flow 是否映射为 DSL message
* Start / End Event 是否映射为 DSL event
* Exclusive / Parallel Gateway 是否映射为 DSL gateway，且类型一致
* BusinessRuleTask 是否映射为 DSL businessrule
* Sequence relation / enabling relation 是否在 DSL flows 中保留
* 不应出现缺失映射或无来源的伪目标元素

### 实现方法

建议采用“**映射契约 + traceability**”方式：

1. 读取 BPMN 模型实例
2. 读取 DSL 模型实例
3. 由转换器输出或补充一份 trace 信息，记录源元素与目标元素对应关系
4. 编写一致性检查脚本，逐项检查：

   * coverage：源元素是否都有目标对应物
   * attribute preservation：关键属性是否保留
   * relation preservation：关系是否保留
   * no spurious target：目标中是否出现无来源业务元素

### 输出结果

输出如下统计：

* 各类 BPMN 元素的覆盖率
* 映射缺失项列表
* 属性不一致项列表
* 关系缺失项列表
* 无来源目标元素列表

### 实验结论含义

这一部分证明的是：

> BPMN 到 DSL 的转换不仅产出了合法 DSL，而且在结构层面保留了源模型中的关键业务元素及其关联关系。

---

## 1C 源目标行为一致性

### 实验目标

验证 BPMN 的控制流、消息流和分支语义，是否在 DSL `flows` 中得到了行为层面的保留。

### 主要验证内容

比较 BPMN 与 DSL 的行为语义表示，重点关注：

* 起始节点集合是否一致
* 后继关系是否一致
* 分支条件是否一致
* 消息协作的方向和启用关系是否一致
* 有界执行路径集合是否一致
* 关键状态迁移是否一致

### 实现方法

建议采用“**统一状态迁移表示 + 有界路径比较**”的方法：

1. 从 BPMN 提取控制流 / 消息流 / 网关分支语义
2. 从 DSL `flows` 提取状态迁移语义
3. 将两者规约为统一的行为表示，例如：

   * 节点
   * 边
   * guard
   * effect
   * state transition
4. 在有界深度下生成路径集合
5. 比较 BPMN 与 DSL 的路径、转移和分支条件是否一致

### 输出结果

输出以下结果：

* 行为图节点/边匹配率
* guard 一致率
* bounded path 一致率
* 差异路径样例
* 状态迁移差异样例

### 实验结论含义

这一部分证明的是：

> DSL `flows` 不是对 BPMN 结构的简单重写，而是在行为语义层面保留了 BPMN 的流程执行逻辑。

# 实验2：DSL→代码静态语义保真验证

实验2聚焦于 **DSL→代码生成** 这一阶段，验证 DSL 的静态语义是否完整落实到两类目标代码中。

---

## 2A DSL→Go 保真

### 实验目标

验证 DSL 中定义的元素和静态语义，是否被完整映射为 Go/Fabric 链码结构。

### 主要验证内容

重点提取并比较：

* participant
* global
* message
* event
* gateway
* businessrule
* oracletask
* handler / trigger
* guard / precondition
* state update
* event emit
* follow-up enable relation

### 实现方法

以 Go AST / IR 为基础，构建 DSL 与 Go 之间的静态语义映射。
建议输出 trace 信息，用于将 DSL rule 映射到 Go function、field、enum、handler 等生成结果。

### 输出结果

输出以下指标：

* DSL 元素覆盖率
* rule 映射覆盖率
* missing mapping 数量
* partial mapping 数量
* 语义偏差项

---

## 2B DSL→Solidity 保真

### 实验目标

验证 DSL 中定义的元素和静态语义，是否被完整映射为 Solidity/Geth 合约结构。

### 主要验证内容

与 Go 侧一致，重点比较：

* 元素定义
* guard / require
* state effect
* lifecycle 相关逻辑
* emitted event
* handler 结构

### 实现方法

以 Solidity AST / IR 为基础，提取结构和语义信息，与 DSL 进行比对。
同样建议生成 trace 文件，用于建立 DSL 到 Solidity 代码节点的可追踪关系。

### 输出结果

输出以下指标：

* DSL 元素覆盖率
* rule 映射覆盖率
* missing mapping 数量
* partial mapping 数量
* 语义偏差项

---

## 2C Go / Solidity 双目标静态差分

### 实验目标

验证同一个 DSL 在两个目标代码上的静态语义实现是否一致。

### 主要验证内容

比较 Go 与 Solidity 中的：

* 可触发 handler 集合
* guard 条件集合
* 状态变量定义
* 状态更新逻辑
* follow-up enable relation
* 终态定义

### 实现方法

将 Go IR 与 Solidity IR 归一化到统一表示后进行差分。
检查两个目标实现是否在静态语义上保持等价。

### 输出结果

输出：

* 双目标语义匹配率
* 单侧缺失项
* guard 差异项
* effect 差异项
* 状态模型差异项

### 实验结论含义

实验2整体证明的是：

> DSL 中定义的静态语义已被完整、可追踪地落实到 Go/Fabric 与 Solidity/Geth 两类目标代码中，且双目标实现之间不存在明显静态语义偏差。

# 实验3：跨平台执行路径一致性实验

实验3聚焦于 **运行时行为的一致性**，验证同一个 DSL 派生出的多平台代码，在实际执行过程中是否与 DSL 参考语义保持一致。

## 实验目标

验证：

* 同一个 DSL 派生出的 Go/Fabric 与 Solidity/Geth 实现
* 在相同合法路径输入下
* 是否产生与 DSL 参考语义一致的执行轨迹
* 且两个平台之间是否一致

---

## 3A DSL 参考轨迹生成

### 目标

基于 DSL `flows` 自动构造参考状态迁移图，并生成若干条合法执行路径及其参考轨迹。

### 实现方法

从 DSL 提取：

* 节点集合
* enable relation
* 分支条件
* parallel join 约束
* 状态推进规则

在给定深度范围内生成路径，并输出每一步的：

* 当前节点
* 当前状态
* guard
* effect
* 预期后继节点
* 终态结果

---

## 3B Fabric / Geth 实际执行

### 目标

在 Fabric 与 Geth 平台上执行同一批路径，获取实际运行轨迹。

### 实现方法

针对每条参考路径构造输入脚本或调用序列，在两个平台分别执行。
记录执行过程中的：

* 当前可触发 handler
* 条件判断结果
* 元素状态变化
* 全局变量变化
* emitted events
* 最终状态

---

## 3C 三方轨迹对比

### 目标

比较以下三者是否一致：

* DSL 参考轨迹
* Fabric 实际执行轨迹
* Geth 实际执行轨迹

### 主要比较维度

包括：

* trigger 是否一致
* guard 是否一致
* state effect 是否一致
* 后继启用集合是否一致
* 终态是否一致
* 非法输入拒绝行为是否一致

### 输出结果

输出：

* 三方轨迹一致率
* Fabric 偏离点
* Geth 偏离点
* 平台间差异点
* 差异路径实例

### 实验结论含义

这一部分证明的是：

> 多平台生成代码不只是静态结构与 DSL 对应，而且在执行层面能够按照 DSL 参考语义进行一致推进，并在 Fabric 与 Geth 两个平台之间保持行为一致。

# 实验4：性能对比实验

实验4关注在行为正确的前提下，两类平台的性能差异。

## 实验目标

比较 Fabric 与 Geth 在相同业务流程负载下的性能表现。

## 主要测试内容

建议选取三类代表性操作：

* 基础流程推进类操作
* 带状态转换的协作操作
* 带业务规则或外部任务触发的复杂操作

## 实现方法

使用 Hyperledger Caliper 构建两类平台的压测场景，分别测量：

* 吞吐量
* 平均时延
* P95 / P99 时延
* 并发扩展性
* 资源消耗情况

## 输出结果

输出：

* Fabric 与 Geth 的吞吐量对比
* 平均时延与尾时延对比
* 并发规模变化下的性能曲线
* 不同类型事务的性能差异

## 实验结论含义

这一部分证明的是：

> 在相同模型驱动生成逻辑下，不同链平台在执行性能上存在差异，进一步体现多链方案在性能与部署适配性方面的比较价值。

# 关于 1C 与实验3 的实现注意：方法上尽量复用

这一点非常关键。
虽然你把 **1C 放在实验1**，把**跨平台运行时一致性**放在实验3，但这两部分在实现时**最好共用同一套语义抽取与路径分析框架**，不要各做一套。

建议统一复用下面四个核心部分。

## 1. 复用统一的语义表示

无论是 1C 还是实验3，都不要直接比较原始 BPMN 图、DSL 文本或平台日志，而是统一规约成同一种中间表示，例如：

* `Node`
* `Transition`
* `Guard`
* `Effect`
* `State`
* `TraceStep`

这样：

* 1C 比较的是 `BPMN semantic graph ↔ DSL semantic graph`
* 实验3 比较的是 `DSL semantic graph ↔ Fabric runtime trace ↔ Geth runtime trace`

同一套表示可以同时服务两组实验。

---

## 2. 复用同一套 DSL 状态迁移图生成器

实验1C里，你需要从 DSL `flows` 提取状态迁移图；
实验3里，你同样需要从 DSL `flows` 生成参考轨迹。

所以这里不要分别实现两个工具，而是直接做成一个公共模块：

* 输入：DSL 模型
* 输出：

  * 状态迁移图
  * 合法路径集合
  * 参考轨迹生成器

这样 1C 用它来和 BPMN 对比，实验3 用它来生成参考执行轨迹。

---

## 3. 复用同一套路径生成策略

建议对 1C 和实验3 统一采用：

* 有界深度
* 忽略循环或限制循环次数
* 覆盖顺序、分支、并行、汇聚等关键路径
* 支持非法输入路径作为补充测试

这样 1C 和实验3 的“路径”概念完全一致。
最理想的做法是：

> 先在 1C 中确认 BPMN 与 DSL 的合法路径集合一致，再直接把 DSL 侧通过验证的路径集复用到实验3中作为平台执行测试集。

---

## 4. 复用统一的轨迹记录格式

实验3需要记录 Fabric/Geth 的运行轨迹；
而 1C 中也需要把 BPMN 与 DSL 的路径展开成“可比较的轨迹形式”。

所以建议统一定义一份轨迹格式，例如每一步都记录：

* step id
* current node
* trigger
* guard result
* state before
* state after
* effect
* enabled successors

这样你后面只需要一套轨迹比较器，就能同时服务：

* BPMN vs DSL
* DSL vs Fabric
* DSL vs Geth
* Fabric vs Geth
