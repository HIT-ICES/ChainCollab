# 实验二上手思路

本文档面向第一次接手 `Experiment/new/exp2_semantic_verification` 的开发者，目标是帮助你快速回答下面几个问题：

- 这个实验二到底在验证什么
- 目录里每个文件和脚本分别干什么
- 一次完整运行的调用链是怎么走的
- 脚本内部是如何实现验证的
- 如果我要手动做一次实验，应该怎么操作
- 实验的输入、生成产物、中间产物分别是什么
- 当前这种设计为什么这样组织


## 1. 实验二的定位

实验二验证的是：

- 一份已经确认可用的 B2CDSL
- 在经过 `newTranslator` 生成 Go 链码和 Solidity 合约后
- 是否还能在目标代码中找到与 DSL 对应的静态结构与控制语义证据

它验证的是：

- 结构一致性
- 控制语义一致性

它不验证的是：

- 链上运行时行为一致性
- Fabric 与 Ethereum 的执行结果是否完全相同
- 业务语义是否真正正确
- OCL 层面的模型合法性

所以实验二的本质不是“跑起来看行为”，而是：

1. 取一份 DSL
2. 生成 Go / Solidity
3. 把 DSL、Go、Solidity 都抽成统一可分析的结构
4. 用断言去检查 DSL 里的元素和控制逻辑是否在代码中有证据


## 2. 当前数据源思路

当前实验二已经切换成：

- 不再自己从 BPMN 重新生成 DSL
- 直接复用 `MDAcheck` 中的正例 DSL

也就是：

- 数据源目录：`/root/code/ChainCollab/src/newTranslator/MDAcheck/bpmn-positive/b2c`

这样设计的原因是：

- 实验一已经在模型合法性层面对这些 DSL 做过筛选
- 实验二应尽量聚焦 “DSL -> 代码” 这一段
- 避免把 “BPMN -> DSL 的不稳定性” 混入实验二
- 能把实验边界切得更清楚

因此现在实验二默认回答的是：

- “对于已经确认的正例 DSL，代码生成阶段是否保留了结构和控制语义”

而不是：

- “从 BPMN 一路到代码整体是否都完全正确”

## 5. 一次完整实验的调用链

总入口是：

- [scripts/run_exp2.py](/root/code/ChainCollab/Experiment/new/exp2_semantic_verification/scripts/run_exp2.py)

它的完整调用顺序如下：

1. 发现 case
   - `discover_case_dirs(cases_dir)`
2. 读取 case 配置
   - `resolve_case(case_root)`
3. 准备 DSL
   - 如果 case 有 `source_b2c` 或 `input.b2c`
   - 直接 `stage_b2c(...)` 复制到结果目录
4. 生成 Go
   - `generate_go(...)`
5. 生成 Solidity
   - `generate_solidity(...)`
6. 解析 DSL
   - 调 `parse_b2c.py`
7. 解析 Go AST
   - 调 `extract_go_ast.go`
8. 导出 Solidity 原始 AST
   - `solc --standard-json`
9. 整理 Solidity AST
   - 调 `extract_sol_ast.js`
10. 做 DSL -> Go 验证
   - 调 `verify_dsl_go_semantics.py`
11. 做 DSL -> Solidity 验证
   - 调 `verify_dsl_sol_semantics.py`
12. 在当前 case 内检查断言表
   - 生成 `case_assertions.json`
13. 汇总所有 case
   - 调 `summary_report.py`


## 6. 运行流程如何走

用更直观的话描述就是：

### 第一步：读 case

脚本只看 `cases/` 下当前启用的目录。

启用条件：

- 目录存在
- 有 `case.json`
- 没有写 `"enabled": false`
- 并且能提供 DSL 来源
  - `source_b2c`
  - 或 `input.b2c`
  - 或 `input.bpmn`

当前主流方式是：

- `case.json` 指向 `MDAcheck` 正例 `.b2c`


### 第二步：准备 DSL

如果 case 提供的是：

- `source_b2c`

那么实验不会自己再生成 DSL，而是直接复制：

- `MDAcheck/.../*.b2c`
  -> `results/cases/<case>/dsl.b2c`

这样设计的原因：

- 固定实验输入
- 保持实验二边界稳定
- 聚焦 “DSL -> 代码”


### 第三步：生成目标代码

利用 `newTranslator` 的现有脚本：

- 生成 Go
- 生成 Solidity

具体在 `common.py` 里通过：

- `nt-go-gen`
- `nt-sol-gen`

把输出复制到：

- `results/cases/<case>/chaincode.go`
- `results/cases/<case>/contract.sol`


### 第四步：把 DSL 变成统一结构

`parse_b2c.py` 会把 DSL 解析成一个统一 JSON：

- `dsl_ast.json`

它会抽出：

- `participants`
- `globals`
- `messages`
- `gateways`
- `events`
- `businessrules`
- `oracletasks`
- `flows`

这样做的理由：

- 验证逻辑不直接依赖原始 `.b2c` 文本
- 后续匹配可以基于统一结构进行


### 第五步：把 Go / Solidity 也变成统一结构

Go：

- `extract_go_ast.go`
- 输出 `go_ast.json`

Solidity：

1. 先跑 `solc --standard-json`
2. 得到 `solc_ast_raw.json`
3. 再用 `extract_sol_ast.js`
4. 输出 `solidity_ast.json`

这样做的理由：

- Go 和 Solidity 的语法结构差异很大
- 先各自解析成 AST
- 再整理成统一分析格式
- 可以避免纯字符串搜索不稳定的问题


### 第六步：单 case 语义验证

Go 侧：

- `verify_dsl_go_semantics.py`

Solidity 侧：

- `verify_dsl_sol_semantics.py`

这两个脚本的工作方式很简单：

1. 读取 `dsl_ast.json`
2. 读取 `go_ast.json` 或 `solidity_ast.json`
3. 对 DSL 中每一类元素逐个调用 `mapping_rules.py` 里的函数
4. 把结果写成单 case 报告

输出：

- `go_semantic_report.json`
- `solidity_semantic_report.json`


### 第七步：批量汇总

最后由 `summary_report.py` 来做汇总。

它会：

1. 读取每个 case 的 `go_semantic_report.json`
2. 读取每个 case 的 `solidity_semantic_report.json`
3. 统计总体覆盖率
4. 再结合 `assertion_table.json`
5. 输出断言覆盖结果


## 7. 脚本内部是如何实现的

### 7.1 `common.py`

它解决的是“实验调度问题”。

主要职责：

- 管路径
- 发现 case
- 读取 case.json
- 准备 DSL 来源
- 调 `newTranslator`
- 调命令执行器

几个关键函数：

- `discover_case_dirs(...)`
  - 扫描有效 case 目录

- `resolve_case(...)`
  - 读取一个 case 的配置

- `stage_b2c(...)`
  - 如果使用现成 DSL，就把它复制到结果目录

- `generate_go(...)`
  - 调用 `nt-go-gen`

- `generate_solidity(...)`
  - 调用 `nt-sol-gen`


### 7.2 `run_exp2.py`

它是总控脚本。

关键函数：

- `process_case(...)`

它串起整个单 case 流程：

1. 准备 `.b2c`
2. 生成 `.go`
3. 生成 `.sol`
4. 生成 `dsl_ast.json`
5. 生成 `go_ast.json`
6. 生成 `solidity_ast.json`
7. 生成 `go_semantic_report.json`
8. 生成 `solidity_semantic_report.json`

最后 `main()` 会遍历所有 case，再调用 `summary_report.py` 出总表。


### 7.3 `parse_b2c.py`

它的核心是把 textX 解析出来的 DSL 模型转成标准 JSON。

设计理由：

- 后续验证不应该直接耦合 textX 运行时对象
- JSON 更适合跨脚本传递
- 也方便人工检查和调试

它会特别把 `flow` 转成统一表示：

- trigger
- conditions
- actions

这对后面验证 `SV08-SV14` 非常关键。


### 7.4 `extract_go_ast.go`

它负责从 Go 代码里抽出验证所需的结构。

例如：

- `functions`
- `calls`
- `assignments`
- `if_conditions`
- `switches`
- `state_fields`
- `string_literals`

设计理由：

- Go 语义验证最核心的是函数、调用、赋值、条件
- 不需要完整保留所有 AST 细节
- 只抽验证需要的最小信息集


### 7.5 `extract_sol_ast.js`

它负责从 `solc` 的原始 AST 中抽出验证需要的信息。

例如：

- `functions`
- `calls`
- `assignments`
- `requires`
- `if_conditions`
- `state_variables`
- `enums`
- `string_literals`

设计理由：

- 原始 Solidity AST 过于庞大
- 不方便直接写验证逻辑
- 需要整理成更接近“语义证据”的表示


### 7.6 `mapping_rules.py`

这是实验二最核心的验证逻辑文件。

它做的事情是：

- 定义“某类 DSL 元素在 Go / Solidity 中算匹配的条件”

例如：

- `verify_global_go(...)`
- `verify_global_sol(...)`
- `verify_message_go(...)`
- `verify_message_sol(...)`
- `verify_businessrule_go(...)`
- `verify_businessrule_sol(...)`
- `verify_flow_go(...)`
- `verify_flow_sol(...)`

每个函数都会返回：

- `matched`
- `evidence`
- `missing_reason`
- `severity`

设计理由：

- 把所有真正的“规则判断”集中到一个文件
- 避免验证逻辑散在多个脚本里
- 以后改断言时更容易定位


### 7.7 `verify_dsl_go_semantics.py` / `verify_dsl_sol_semantics.py`

这两个脚本是 `mapping_rules.py` 的包装器。

它们主要负责：

- 按 DSL 元素遍历
- 调用对应验证函数
- 组装统一 report

输出格式统一成：

- `elements`
- `summary`

这样 `summary_report.py` 才能无差别汇总。


### 7.8 `summary_report.py`

它负责两层汇总：

1. case 级汇总
   - `exp2_summary.md`
   - `exp2_summary.json`
2. 断言级汇总
   - `assertion_coverage.md`
   - `assertion_coverage.json`

它不是重新做元素验证，而是：

- 复用单 case report
- 再按断言表做聚合

设计理由：

- 元素验证和断言统计分层
- 修改断言时不一定要重写底层验证器
- 只要证据足够，就能重算覆盖


## 8. 输入、生成产物、中间产物

### 8.1 输入

当前主输入是：

- `cases/<case>/case.json`
- `source_b2c` 指向的 `.b2c`

可能的输入类型有三种：

1. `source_b2c`
   - 当前默认方式
2. `input.b2c`
   - 手工放入 DSL 也可以
3. `input.bpmn`
   - 兼容旧流程，但当前默认不用


### 8.2 生成产物

每个 case 的最终主产物包括：

- `<case>.b2c`
- `<case>.go`
- `<case>.sol`
- `go_semantic_report.json`
- `solidity_semantic_report.json`


### 8.3 中间产物

每个 case 的中间产物包括：

- `dsl_ast.json`
- `go_ast.json`
- `solc_ast_raw.json`
- `solc_ast_input.json`
- `solidity_ast.json`

这些中间产物非常重要，因为它们正是排查问题时最直接的依据。

例如：

- 看 DSL 是否被正确解析
  - `dsl_ast.json`
- 看 Go 是否抽到了 `ChangeMsgState`
  - `go_ast.json`
- 看 Solidity 是否抽到了 `inst.stateMemory.xxx = ...`
  - `solidity_ast.json`


## 9. 为什么设计成现在这样

### 9.1 为什么不用直接 grep

因为纯文本搜索不稳定：

- 容易误匹配
- 很依赖命名风格
- 很难表达条件结构和赋值关系

所以采用：

- DSL -> 统一 JSON
- Go / Solidity -> 统一 AST JSON
- 再做规则匹配


### 9.2 为什么把规则集中在 `mapping_rules.py`

因为验证逻辑最容易频繁调整。

集中管理的好处是：

- 断言改动时更容易同步
- 容易排查到底是匹配规则问题还是生成器问题
- 便于单点修复


### 9.3 为什么断言覆盖放在 `summary_report.py`

因为“元素匹配成功”不等于“断言覆盖完成”。

断言覆盖需要：

- 按 `element_type` 聚合
- 按 `targets` 聚合
- 按 `mode` 聚合
- 按正负例分别统计

这更像汇总层逻辑，而不是底层元素验证逻辑。


### 9.4 为什么现在直接使用 MDAcheck 正例 DSL

原因有 4 个：

- 避免 BPMN -> DSL 这一段影响实验二
- 保证输入 DSL 是稳定的、已确认的
- 更适合单独观察 DSL -> 代码的保真情况
- 让实验一和实验二的边界更清楚

## 11. 我排查问题时应该先看什么

推荐顺序：

1. 看 `run_log.json`
   - case 是成功还是失败
2. 看对应 `error.log`
   - 如果失败，先看卡在哪一层
3. 看 `dsl_ast.json`
   - DSL 是否被正确抽出来
4. 看 `go_ast.json` / `solidity_ast.json`
   - 目标代码证据是否存在
5. 看 `go_semantic_report.json` / `solidity_semantic_report.json`
   - 单 case report 里 evidence 是否合理
6. 最后再看 `assertion_coverage.md`
   - 断言层是不是被正确汇总

不要一上来只看 `exp2_summary.md`。

因为：

- `exp2_summary.md` 是最上层结果
- 真正的问题通常发生在更底层中间产物


## 12. 当前已知注意点

### 12.1 `amazon` case

当前 `amazon` 在 Solidity 解析阶段会失败。

原因是：

- 生成的 Solidity 中使用了保留字 `type` 作为参数名

所以如果你看到 `amazon` 没出现在成功汇总里，这是预期内的已知问题。


### 12.2 `SV07`

当前 `SV07` 仍可能显示 `unobserved`。

原因不是脚本错误，而是：

- 当前正例 DSL 集合中没有 `oracletask`


## 13. 一句话上手建议

如果你是第一次接手，最推荐的理解路径是：

1. 先看 `README.md`
2. 再看一个具体 case 的 `results/cases/<case>/`
3. 然后看 `run_exp2.py`
4. 再看 `mapping_rules.py`
5. 最后看 `summary_report.py`

因为实验二真正的核心不是“怎么调用脚本”，而是：

- DSL 被怎么抽象
- 代码被怎么抽象
- 两者是如何通过规则被对齐的

理解了这三点，整个实验二就基本吃透了。
