# Relayer 实验设计方案（可用性与有效性验证）

## 1. 目标与研究问题

目标：验证“跨链中继合约 + 链下 Relayer 节点”方案在工业物联网场景下的可用性、可靠性与性能。

核心研究问题：

1. 可用性：在节点故障、网络抖动下，系统能否持续完成跨链任务？
2. 正确性：是否能保证“目标链只处理一次”（防重复）？
3. 性能：跨链延迟和吞吐在不同负载下如何变化？
4. 健壮性：面对异常输入（重复、伪造、越权）时，是否能正确拒绝？
5. 成本：单条跨链任务的链上 gas 成本与链下资源开销如何？

---

## 2. 待验证系统（SUT）

建议最小系统组成：

1. 源链合约 `SourceRelayAdapter`：发布跨链任务事件（taskId, payloadHash, nonce, targetChain）。
2. 目标链合约 `TargetRelayAdapter`：接收并执行任务，维护去重状态（processed[taskId]）。
3. Relayer 节点服务（可多副本）：监听源链事件 -> 验证 -> 投递目标链 -> 回执记录。
4. 持久化存储（本地 DB/Redis）：记录投递状态、重试次数、幂等键。

工业场景映射（建议）：

- 设备告警（高优先级、低负载）
- 产线状态同步（中频）
- 能耗汇总上链（批量）
- 质检结果跨链流转（中频、强一致要求）

---

## 3. 实验指标定义

## 3.1 可用性与正确性

- `SuccessRate = 成功跨链任务数 / 总任务数`
- `ExactlyOnceRate = 无重复执行任务数 / 成功任务数`
- `InvalidRejectRate = 被正确拒绝的非法请求数 / 非法请求总数`
- `RecoveryTime`：故障注入后恢复到稳定吞吐所需时间

## 3.2 性能

- 端到端延迟（源链确认到目标链确认）：`p50/p95/p99`
- 吞吐：`TPS`（每秒完成跨链任务数）
- 积压深度：队列 backlog 峰值与恢复曲线

## 3.3 成本

- 源链每任务 gas（事件发布）
- 目标链每任务 gas（验证+执行）
- 总 gas / task
- Relayer CPU/内存占用（平均/峰值）

---

## 4. 实验矩阵（建议 5 组）

## E1 功能与基线可用性

- 目的：验证基本链路可用
- 方法：单 Relayer、稳定网络、低负载（10/50/100 task）
- 输出：成功率、重复率、延迟基线

## E2 多负载性能实验

- 目的：评估扩展能力
- 方法：并发强度梯度（50, 100, 200, 500 req/s）
- 输出：TPS、p95/p99 延迟、积压曲线

## E3 故障恢复实验

- 目的：验证高可用能力
- 故障注入：
  1. 单 relayer 进程 kill
  2. 目标链 RPC 暂时不可用
  3. 网络抖动（延迟/丢包）
- 输出：RecoveryTime、失败重试成功率、任务最终成功率

## E4 幂等与重放攻击实验

- 目的：验证防重复执行能力
- 注入场景：
  1. 重复广播同一 taskId
  2. Relayer 重启后重复投递
  3. 重放旧 nonce/task
- 输出：ExactlyOnceRate、重复抑制率

## E5 安全拒绝实验

- 目的：验证非法请求处理
- 注入场景：
  1. 非白名单 Relayer 提交
  2. payloadHash 不匹配
  3. 签名错误/伪造
- 输出：InvalidRejectRate、误拒率（合法请求被拒）

---

## 5. 工作负载与数据设计

建议使用“工业事件任务集”（JSON）驱动实验，字段示例：

```json
{
  "taskId": "uuid",
  "type": "alarm|energy|quality|status",
  "priority": 1,
  "payload": { "deviceId": "M-01", "value": 83.2, "ts": 1730000000 },
  "targetChain": "chainB"
}
```

建议准备三类负载：

1. 平稳负载（均匀分布）
2. 突发负载（泊松突刺）
3. 混合负载（80%正常 + 20%高优任务）

## 7. 最小落地路线（建议顺序）

1. 实现 `SourceRelayAdapter` / `TargetRelayAdapter`（含 taskId 去重）
2. 实现单节点 relayer（监听+投递+状态机）
3. 先跑 E1，拿到第一版成功率与延迟基线
4. 加入重试/幂等后跑 E4
5. 加入故障注入脚本后跑 E3
6. 最后做 E2/E5，汇总总报告

---

## 8. 验收标准（建议）

可作为阶段性目标：

1. 成功率 `>= 99%`（E1/E2）
2. ExactlyOnceRate `>= 99.99%`（E4）
3. 非法拒绝率 `>= 99%`（E5）
4. 故障恢复时间 `<= 60s`（E3）

达到以上指标，可较有力支撑“方案可用性与有效性”结论。
