## 1. 这一部分在论文中的定位

建议将这部分内容定位为：

- 面向以太坊兼容环境的链下决策执行支持方案；
- 对已有 BlockCollab / IBC 联盟链执行路线的补充扩展；
- 面向“复杂规则不能直接上链”这一问题的可信预言机方案。

核心不是强调“重新做了一套区块链系统”，而是强调：

- 复杂 DMN 决策逻辑仍在链下执行；
- 区块链负责请求发起、结果记录、结果校验和最终确认；
- Chainlink OCR 负责把链下结果以多节点聚合的方式可信引入链上。

## 2. 可以写的研究动机

这一部分可以从三个矛盾切入：

1. 区块链智能合约要求执行确定性，但业务决策常依赖链外数据和复杂规则。
2. DMN 规则适合在链下引擎中执行，但链下执行天然缺少链上可验证性。
3. 单节点预言机虽然能把链下结果传回链上，但会形成新的信任中心。

因此需要一种机制，同时满足：

- DMN 继续在链下执行；
- 结果能被链上合约验证；
- 结果接入不依赖单一预言机节点。

## 3. 总体方案概括

可将整体描述为“链上请求、链下计算、多节点聚合、链上校验、结果定稿”的混合执行架构。

建议在论文中明确四层结构：

- 区块链执行层：Geth 私有链 + DMN 请求合约 + OCR Aggregator。
- 预言机协同层：Chainlink directrequest + OCR 多节点网络。
- DMN 计算缓存层：本地 DMN 服务，负责执行和缓存结果。
- 结果确认层：合约比对 baseline 与 OCR 聚合值，并执行 finalize。

## 4. 与 README/代码实现对应的关键组件

### 4.1 链上部分

- `MyChainlinkRequesterDMN.sol`
- `OffchainAggregator`
- `Operator`

业务入口合约的职责可以概括为：

- 发起 DMN 请求；
- 保存 baseline 原始结果和哈希；
- 校验 OCR 聚合值是否匹配；
- 输出最终确认结果。

### 4.2 Chainlink 任务部分

- `job-spec-dmn-event.toml`
- `job-spec-ocr-dmn.toml`
- `job-spec-ocr-writer.toml`

其中可以强调两类任务：

- `directrequest` 任务：监听合约请求，调用 DMN 服务计算并回写 baseline。
- `OCR` 任务：从缓存读取 `hashDec`，参与多节点 observation 聚合。

### 4.3 DMN 服务部分

关键接口：

- `/api/dmn/calc`
- `/api/dmn/latest`
- `/api/dmn/by-hash`
- `/api/dmn/ack`

DMN 服务的作用不是单纯“算一次结果”，而是承担：

- 计算；
- 缓存；
- 哈希映射；
- 按 hash 回查原始结果；
- 在链上写回完成后收敛状态。

## 5. 最值得写清楚的核心机制

这部分建议作为正文重点，因为它最能体现方案设计，不只是工程堆叠。

### 5.1 双通道机制

整套方案不是直接让 OCR 上传原始 DMN 结果，而是分成两条通道：

- baseline 通道：`directrequest` 将 `raw` 写回合约；
- OCR 通道：多节点只上传 `hashDec` 并聚合。

这个设计的意义是：

- 原始结果 `raw` 能保留审计依据；
- OCR 只处理数值，简化 observation；
- 合约只做一致性校验，不承担复杂 DMN 求值。

### 5.2 三类结果对象

DMN 服务计算后会得到三类值：

- `raw`：原始结果文本；
- `hash`：`keccak256(raw)`；
- `hashDec`：`hash` 的低 128 位十进制表示。

可以把它解释为一组“链下表达 - 链上映射”关系：

- `raw` 负责业务语义表达；
- `hash` 负责唯一标识；
- `hashDec` 负责适配 OCR 数值上报。

### 5.3 链上最终确认逻辑

业务合约的确认逻辑可以概括为：

1. 保存 baseline；
2. 接收 OCR 聚合值；
3. 执行 `isOcrMatch(requestId)`；
4. 若匹配，则 `finalize`；
5. 对外通过 `getFinalizedRaw(requestId)` 暴露最终结果。

这里要突出一个点：

- 链上并不直接信任链下服务；
- 链上也不直接信任某一个 Chainlink 节点；
- 链上信任的是“baseline + 多节点 OCR 聚合 + 合约一致性判断”这个组合机制。

## 6. 执行流程可以怎么写

正文里可以按下面这个顺序展开：

1. 业务侧调用 `requestDMNDecision()`。
2. 合约发出 `OracleRequest`。
3. directrequest job 监听事件并调用 `/api/dmn/calc`。
4. DMN 服务执行决策，生成 `raw/hash/hashDec` 并缓存。
5. directrequest job 调用 `commitBaselineFromRaw(requestId, raw)` 写 baseline。
6. OCR 节点从 `/api/dmn/latest?requireReady=1` 获取 `hashDec`。
7. OCR 网络聚合后将结果写入 Aggregator。
8. 监听器捕获 `NewTransmission`。
9. 触发 `finalizeWithOcrAnswer()` 或 `finalize()`。
10. 合约校验通过后输出 `getFinalizedRaw(requestId)`。
11. DMN 服务收到 ACK，清理缓存。

## 7. 一致性与可信性可以怎么论述

这里建议分四点写。

### 7.1 链下计算、链上确认

强调原因：

- DMN 规则复杂；
- 输入数据可能来自企业系统；
- 直接链上执行成本高且维护困难。

结论：

- 将 DMN 执行保留在链下；
- 将结果确认与审计放在链上。

### 7.2 baseline 与 OCR 分离

强调这不是冗余设计，而是职责拆分：

- baseline 负责保留原始业务结果；
- OCR 负责提供多节点一致性证明。

### 7.3 哈希低 128 位映射

这部分是实现细节，但论文里最好说明：

- 这是为了适配 OCR observation 的数值格式；
- 服务端、Job、合约三端必须保持完全一致；
- 这是当前工程实现中的一种折中设计。

如果你想写得更严谨，可以顺带指出：

- 低位截断带来理论碰撞风险；
- 当前原型在受控实验环境下接受这一工程折中；
- 后续可扩展为更强的编码与校验机制。

### 7.4 ACK 收敛机制

这一点建议写，因为它体现了你考虑了状态管理问题。

可以写成：

- 链上写回完成后，如果缓存不清理，后续请求可能读到旧值；
- 因此服务端引入 `requestId + block timestamp` 双条件收敛机制；
- 这样能避免并发请求或连续请求时误删新结果。

### 引言

在复杂业务场景下，多方协作规则不仅需要被形式化表达，还需要被稳定地映射为可执行、可验证和可审计的运行环境。与单一组织内部流程不同，多方协作通常同时涉及跨主体身份管理、链上状态共享、链外业务系统调用、外部事实接入以及决策结果确认等多个环节。这意味着，仅依靠单一链上智能合约并不足以支撑真实业务中的协作执行，还需要构建覆盖链上与链下、流程与决策、数据与规则的统一执行环境。

已有研究与本文前期工作已经在基于 Hyperledger Fabric 的多方协作流程执行方面取得了一定进展，例如基于 BPMN Choreography 与决策模型的协作框架能够为联盟链场景提供较强的隐私保护与组织控制能力。然而，从复杂业务的一般执行需求看，仅支持联盟链并不足以覆盖所有协作场景。一方面，开放协作或跨平台协作场景还需要兼容以太坊等 EVM 执行环境；另一方面，链上执行模型本身要求智能合约具备确定性，而现实业务中的外部数据获取、复杂规则求值和 DMN 决策执行往往天然发生在链下。因此，如何通过预言机将链下计算结果以可信方式引入区块链，成为多方可信协作执行环境中的关键问题。

基于上述考虑，本章面向复杂业务中的链上链下协同执行需求，研究一种兼容异构区块链的多方可信协作执行环境，并进一步给出基于 Geth 与 Chainlink 的 DMN 链下计算支持方案。该方案将 DMN 决策求值保留在链下执行，同时利用 Chainlink OCR 网络对链下结果进行多节点聚合与链上确认，从而在不破坏链上确定性约束的前提下，将复杂业务决策结果安全地纳入协作流程执行过程中。相较于仅在联盟链内部完成决策与流程支撑的方案，这一设计进一步扩展了本文研究在权限链与以太坊兼容链上的统一适用性，也为后续跨链协作和异构执行环境协同提供了基础。

### 3.1 基于异构区块链的多方可信协作执行环境

#### 3.1.1 设计目标与扩展动机

面向复杂业务的多方协作执行环境，首先需要解决的是协作规则如何在不同类型区块链平台上获得一致的执行承载能力。联盟链环境具有成员可控、权限边界明确和数据保护能力较强等特点，适合较为稳定的组织协作场景；以太坊兼容环境则提供了统一的 EVM 执行接口、更成熟的智能合约工具链以及更丰富的预言机生态，适合研究链上链下协同和开放式组件集成问题。由于两类环境在共识机制、部署形态、预言机支持方式以及链下交互模式上存在显著差异，若要构建真正面向复杂业务的执行底座，就不能将研究范围局限于单一区块链平台。

从决策执行角度看，DMN 模型虽然能够有效支持业务决策与流程分离，但其执行仍面临两方面约束。其一，DMN 规则结构复杂、输入参数类型丰富，若直接将 DMN 引擎移植到链上，会导致智能合约实现复杂、Gas 成本高且可维护性差。其二，决策输入往往依赖外部系统、链外文档或实时业务数据，而这些信息无法直接被区块链确定性执行模型原生获取。已有研究尝试在以太坊环境下直接执行 DMN 决策或将决策逻辑转写为链上代码，但在复杂业务场景下，这种方式难以兼顾灵活性、可扩展性与工程可行性。

因此，本文将链下计算支持问题定义为：在保留区块链可信记录与可验证性的同时，将复杂 DMN 决策从链上执行中解耦出来，并通过多节点预言机协作实现结果的可信引入。围绕这一目标，本章进一步构建基于 Geth 私有链、Chainlink OCR 网络以及本地 DMN 服务的混合执行环境。其基本思想是：由 Geth 私有网络作为智能合约与聚合结果的可信承载层，由 Chainlink 作为链上与链下之间的桥接层，由 DMN 服务作为复杂业务规则的实际执行层，从而形成一个可支持“链上请求、链下计算、链上确认”的异构协同架构。

#### 3.1.2 基于 Geth 与 Chainlink 的 DMN 链下计算架构

基于上述目标，本文设计了一种面向以太坊兼容环境的 DMN 链下计算架构。该架构由区块链执行层、预言机协同层、DMN 计算缓存层和结果确认层四部分组成，可概括为：业务合约在 Geth 网络中发起 DMN 请求，Chainlink 多节点监听该请求并调用本地 DMN 服务完成链下计算，随后一部分结果被写入链上基准，另一部分结果以 OCR 观测值形式参与聚合，最终由业务合约完成结果校验与定稿。

区块链执行层以 Geth 私有链为基础，主要部署四类合约：LINK 代币合约、Operator 合约、OCR Aggregator 合约以及 DMN 请求合约。其中，DMN 请求合约 `MyChainlinkRequesterDMN.sol` 是业务入口，它负责接收业务系统提交的 `dmnContent`、`decisionId` 和 `inputData`，通过 `requestDMNDecision()` 触发链上请求，并在后续阶段保存基准结果、校验 OCR 聚合结果以及输出最终结果。

预言机协同层由一个 bootstrap 节点和多个 Chainlink 从节点组成，形成 OCR 网络。该层包含两类关键任务。第一类是 directrequest 监听任务，对应 `job-spec-dmn-event.toml`，其职责是监听业务合约触发的 `OracleRequest` 事件，解析请求参数后调用 DMN 服务接口 `/api/dmn/calc`，再将返回的原始结果通过 `commitBaselineFromRaw(requestId, raw)` 写回业务合约。第二类是 OCR 观测任务，对应 `job-spec-ocr-dmn.toml`，其职责是从各节点本地缓存接口 `/api/dmn/latest?requireReady=1` 读取最近一次决策结果对应的 `hashDec` 数值，并将其作为 OCR 的 observation 参与链下共识与链上报告提交。

DMN 计算缓存层由本地 CDMN 服务承担。该服务并不直接暴露给业务合约，而是作为 Chainlink 节点可访问的链下决策执行组件。其核心接口包括：`/api/dmn/calc`，用于执行 DMN 规则并生成原始结果；`/api/dmn/latest`，用于提供最近一次缓存结果给 OCR 节点读取；`/api/dmn/by-hash`，用于通过哈希查回原始结果；以及 `/api/dmn/ack`，用于在链上写回完成后清理缓存。服务端在执行 DMN 决策后，会将结果整理为 `raw`、`hash` 和 `hashDec` 三个互相关联的对象。其中，`raw` 表示原始决策结果，`hash` 表示对 `raw` 执行 `keccak256` 后得到的哈希值，而 `hashDec` 则表示该哈希低 128 位的十进制形式，用作 OCR 聚合输入。

结果确认层负责将链下执行结果转化为链上可验证状态。该层不是简单地将 OCR 聚合结果直接作为最终结果，而是采用“基准写入 + OCR 聚合 + 合约校验 + 结果定稿”的四阶段结构。具体而言，directrequest 任务首先将 `raw` 写入业务合约，形成链上基准结果；随后 OCR 网络只聚合 `hashDec`，并将聚合值写回 OCR Aggregator；业务合约再通过 `isOcrMatch(requestId)` 判断 OCR 聚合结果与基准哈希是否一致；只有在一致条件满足后，监听器才会触发 `finalizeWithOcrAnswer()` 或 `finalize()`，将结果正式固化为可对外查询的最终状态。这样，链上既保留了与原始结果对应的审计依据，又通过多节点聚合降低了单点预言机带来的可信性风险。

#### 3.1.3 核心执行流程

该架构的核心执行流程可以描述为以下八个阶段。

首先，业务系统或上层协作流程调用 DMN 请求合约的 `requestDMNDecision()` 方法，提交 DMN 模型内容、目标决策标识和输入参数。合约收到请求后发出 `OracleRequest` 事件，并进入等待外部结果的状态。

其次，Chainlink 节点上的 directrequest 任务捕获该事件，解析出请求中携带的 `dmnContent`、`decisionId` 和 `inputData`，并将其转发给本地 DMN 服务的 `/api/dmn/calc` 接口。DMN 服务完成链下求值后，生成标准化原始结果 `raw`，同时计算其哈希 `hash` 和低 128 位十进制映射 `hashDec`，并将这些内容缓存于本地服务中。

再次，directrequest 任务以 `raw` 为输入调用业务合约的 `commitBaselineFromRaw(requestId, raw)` 方法，将本轮请求对应的基准结果写入链上。该步骤的作用并非直接完成业务结果确认，而是在链上形成一个与原始文本一一对应的可信基准，为后续 OCR 聚合结果校验提供参照。

随后，OCR 网络中的多个 Chainlink 节点周期性地从各自对应的 DMN 服务缓存接口 `/api/dmn/latest?requireReady=1` 读取 `hashDec` 数值。该数值作为每个节点的 observation 参与 OCR 共识协议，经过链下报告聚合后，由 OCR Aggregator 合约统一写回 Geth 网络。由于 OCR 只处理数值型 observation，避免了在链上或 OCR 报文中直接承载长文本结果，从而降低了报文复杂度与链上处理成本。

在 OCR 聚合结果写回后，结果确认层开始工作。业务合约根据 `requestId` 取出前期保存的基准哈希，并将其与 OCR Aggregator 上的最新聚合值进行比较。如果两者一致，则说明多节点预言机网络对链下计算结果形成了与基准结果相同的确认意见，合约便允许进入结果定稿阶段。若两者不一致，则请求结果不会被确认，从而在链上形成对异常或不一致结果的拒绝机制。

之后，OCR 写回监听器会监听 Aggregator 合约产生的 `NewTransmission` 事件，并触发对应的 webhook 任务，最终调用 `finalizeWithOcrAnswer(requestId, hashLow)` 或 `finalize(requestId)`。业务合约在该方法内部再次执行一致性校验，确保只有匹配成功的结果才能被固化。定稿完成后，业务侧即可通过 `getFinalizedRaw(requestId)` 读取最终 DMN 决策结果。

最后，为了避免旧缓存影响后续请求，DMN 服务会在确认 OCR 写回完成后执行 ACK 清理逻辑。监听器在上报 ACK 时会携带链上区块时间戳，服务端仅在缓存写入时间不晚于该时间戳时才清理对应记录，从而避免并发或连续请求场景下误删更晚生成的新缓存结果。

#### 3.1.4 一致性与可信性保障机制

该方案的关键不在于把 DMN 本身搬到链上，而在于通过结构化的分层机制，在链下灵活计算和链上可信确认之间建立稳定连接。为此，本文从以下四个方面保证执行的一致性与可信性。

第一，采用链下计算、链上确认的职责分离机制。DMN 决策规则保留在链下服务中执行，从而避免将复杂规则引擎直接嵌入智能合约造成的成本与维护负担；而链上则只负责记录请求、存储基准哈希、接收 OCR 聚合值以及完成一致性判断。这样既保留了区块链的可验证性，也保持了 DMN 模型的灵活更新能力。

第二，采用基准写入与 OCR 聚合分离机制。该方案并不让 OCR 网络直接上传原始文本结果，而是由 directrequest 任务先把 `raw` 写成链上 baseline，再让 OCR 网络只对 `hashDec` 做多节点聚合。这样设计有两个优势：一是原始结果仍保留可追溯路径，便于审计与复现实验；二是 OCR 网络处理对象固定为数值，降低了 observation 格式复杂度，提高了聚合稳定性。

第三，采用统一哈希映射机制。系统规定 OCR 上报值必须是 `keccak256(raw)` 的低 128 位十进制表示。该约束使得 DMN 服务端、Chainlink job 和业务合约之间形成了统一的数据解释方式。虽然低位截断在理论上引入了碰撞风险，但在受控实验环境和业务原型场景下，该设计在计算便利性与验证成本之间取得了较好的折中。论文中可将这一点明确表述为当前实现的工程化取舍，并为后续扩展到更强哈希编码或多字段校验保留空间。

第四，采用基于请求标识与时间戳的缓存收敛机制。DMN 服务既支持按 `requestId` 精确清理缓存，也支持根据 OCR 回写区块时间戳判断是否应清理当前最新缓存。该机制避免了旧缓存覆盖新请求结果的问题，使得本方案在多轮请求连续触发时仍能保持较好的状态收敛性。

#### 3.1.5 与已有 BlockCollab 工作的关系

从研究定位看，本文前期的 BlockCollab / IBC 工作主要面向 Hyperledger Fabric 等联盟链环境，重点解决基于 BPMN Choreography 的多方协作流程建模、智能合约生成和链上链下混合执行问题。该类工作强调组织间协作流程在权限可控环境中的隐私保护、部署治理与协同执行，适合较稳定的企业间合作场景。

本节提出的 Geth + Chainlink + DMN 方案则面向以太坊兼容环境，重点解决复杂决策规则如何在保持链上确定性的前提下，通过多节点预言机机制被可信引入链上执行过程的问题。两者并不是替代关系，而是处于同一研究主线下的互补扩展关系。前者提供了基于联盟链的流程执行底座，后者则进一步验证了本文方法在 EVM 环境与预言机场景下的可迁移性，并对论文中“兼容异构区块链与可信预言机”的研究目标形成更完整的工程支撑。

更重要的是，BlockCollab 论文在结论中已经指出，未来需要进一步支持权限链之外的区块链环境，并增强预言机设计。本文这里给出的方案恰好沿着这一扩展方向展开：一方面，它将协作决策执行底座从 Fabric 延伸到 Geth 私有链；另一方面，它将预言机从一般的链外调用组件细化为包含 directrequest、OCR 聚合、结果比对和 ACK 收敛机制的完整执行链路。因此，该工作可以视为对已有论文原型系统和方法体系在“以太坊兼容执行环境 + DMN 链下计算 + 多节点预言机聚合”方向上的实质性延展。

### 3.2 基于 BPMN 编排的协作流程构建与执行方法

在多方协作流程层面，前述 DMN 链下计算方案可以被视为对 BPMN 业务规则任务的一种底层执行支撑。对于集成 BPMN 与 DMN 的协作流程而言，流程控制流仍由 BPMN Choreography 或协作编排模型负责描述，而决策节点则不必直接在链上编码为复杂条件判断，而是可以映射为对 `requestDMNDecision()` 的调用。换言之，流程层关注“何时需要做决策、由谁触发决策以及决策结果如何影响流程推进”，而 DMN + Chainlink 机制负责“如何以可信方式获得该决策结果”。

由此，在面向复杂业务的执行方法中，流程构建与执行可形成如下协同关系：首先，建模层利用 BPMN 描述多方消息交互、任务衔接和分支收敛；其次，在涉及复杂规则判定的节点处，将决策逻辑抽象为可独立演化的 DMN 模型；最后，在执行层通过以太坊智能合约与 Chainlink 预言机网络将 DMN 结果安全引入链上状态机。这样，协作流程不再需要将所有决策细节硬编码到流程网关或智能合约分支中，从而增强了流程与决策分离建模、异构执行环境适配以及业务规则灵活更新的能力。

### 本章小结

本章围绕复杂业务中的多方可信协作执行需求，构建了一个面向异构区块链的链上链下协同执行框架，并重点给出了基于 Geth、Chainlink OCR 与 DMN 服务的链下计算支持方案。该方案通过引入 Geth 私有链作为 EVM 执行底座、Chainlink 多节点网络作为链上链下桥接机制以及 DMN 服务作为复杂决策求值组件，形成了从链上请求、链下执行、多节点聚合、链上校验到结果定稿的完整执行链路。

与将复杂规则直接写入链上合约的做法相比，本章方法将决策计算与结果确认解耦，并通过基准写入、OCR 聚合、一致性比对和缓存 ACK 收敛等机制，在保持区块链确定性执行约束的前提下，实现了对链外决策结果的可信接入。与此同时，该方案也将本文已有面向联盟链的多方协作执行研究进一步扩展到了以太坊兼容环境，为后续跨链协同、异构执行环境融合以及更复杂的协作流程落地提供了方法与工程基础。
