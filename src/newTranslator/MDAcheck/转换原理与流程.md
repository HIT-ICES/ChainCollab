# `.b2c → .xmi` / `b2c.tx → b2c.ecore` 转换原理与流程

本目录的目标是把 **B2CDSL（textX DSL）** 接入 **EMF/Ecore + Eclipse OCL** 的校验链路：

- `b2c.tx`（语法/元模型定义，textX）  
  → 导出为 `b2c.ecore`（EMF 元模型，Ecore）
- `*.b2c`（DSL 实例文本）  
  → 转换为 `*.xmi`（EMF 实例模型，XMI）
- `check.ocl`（OCL 约束）  
  → 在 Eclipse/EMF OCL 环境中对 `*.xmi` 做约束校验

对应脚本：

- 元模型导出：`export_b2c_ecore.py`
- 实例转换：`b2c_to_xmi.py`

---

## 0. 背景：textX 与 EMF 的“模型”分别是什么

### 0.1 textX 侧

- `b2c.tx` 定义了语法规则，textX 运行时会生成一组 **Python 类**（如 `Contract`/`Message` 等），并附带 `_tx_attrs` 描述属性/引用/多重性/包含关系等元信息。
- `*.b2c` 会被解析成一棵 **Python 对象图**（实例模型）。

### 0.2 EMF 侧

- `.ecore` 是元模型文件：定义 `EPackage/EClass/EAttribute/EReference/EEnum` 等。
- `.xmi` 是实例模型文件：把 `EObject`（由 `.ecore` 定义的类型）序列化到 XML。
- Eclipse OCL 通常在 **Ecore 元模型 + XMI 实例模型** 的上下文中执行 OCL 约束。

因此：想在 Eclipse 里校验 `.b2c`，一般需要先把 `.b2c` 变成 `.xmi`。

---

## 1) `export_b2c_ecore.py`：从 `b2c.tx` 导出 `b2c.ecore`

### 1.1 输入/输出

- 输入：`b2c.tx`（默认 `../DSL/B2CDSL/b2cdsl/b2c.tx`）
- 输出：`b2c.ecore`（默认写到 `MDAcheck/b2c.ecore`）

运行：

```bash
./.venv/bin/python MDAcheck/export_b2c_ecore.py
```

也可显式指定：

```bash
./.venv/bin/python MDAcheck/export_b2c_ecore.py --tx path/to/b2c.tx --out path/to/b2c.ecore
```

### 1.2 核心思路：利用 textX 的元信息“重建”一份 Ecore

textX 在加载 `b2c.tx` 后，会在 `mm.namespaces['b2c']` 中提供规则对应的类（或 match rule/abstract rule）。

脚本的主流程是：

1. `metamodel_from_file(b2c.tx)` 得到 `TextXMetaModel`
2. 找到 `b2c` 命名空间里所有规则对应的“类/类型”
3. 生成一个 `EPackage(name='b2c', nsURI=..., nsPrefix='b2c')`
4. 将“枚举类规则”和“普通类规则”分别映射成 `EEnum` 与 `EClass`
5. 把每个类的 `_tx_attrs` 映射成 `EAttribute` 或 `EReference`
6. 通过 `ResourceSet + resource.save()` 写成 `.ecore`

### 1.3 枚举（EEnum）的来源与处理

在 `b2c.tx` 里类似下面的规则：

```text
ElementState:
  'INACTIVE' | 'READY' | 'PENDING_CONFIRMATION' | 'DONE'
;
```

在 textX 里它属于 “match rule”（`_tx_type == 'match'`），本质是一组字面量。

脚本里用 `ENUM_RULES` 指定哪些规则应当被映射为 EEnum：

- `ScalarType`
- `ElementState`
- `GatewayType`
- `MessageCondition`
- `RuleCondition`

实现方式：

- 读取 `b2c.tx` 的文本
- `_extract_enum_literals()` 用正则提取 `'...'` 中的字面量并保持顺序
- 为每个枚举创建 `EEnum(enum_name, literals=[...])`

注意：`RelationOperator` 包含 `==/!=/>=` 这类不适合作为 `EEnumLiteral.name` 的符号，因此脚本刻意不把它做成 `EEnum`，而是让相关字段在 Ecore 中变成 `EString`（更稳，也更通用）。

### 1.4 类（EClass）与继承（eSuperTypes）

在 textX 元模型里：

- 抽象规则（如 `FlowElement`/`FlowItem` 等）会是 `_tx_type == 'abstract'`
- 这些抽象规则在 `_tx_inh_by` 中列出子类（如 `FlowElement` 被 `Message/Gateway/Event/BusinessRule` 继承）

脚本做的事情：

- 遍历命名空间中所有非 `match` 的规则，建立 `EClass(name, abstract=...)`
- 对每个抽象类，把其 `_tx_inh_by` 的子类 `EClass` 添加 `eSuperTypes`

这样导出的 `b2c.ecore` 就能表达：`Message` 是 `FlowElement` 的子类、`StartFlow` 是 `FlowItem` 的子类等。

### 1.5 属性/引用：`_tx_attrs` → `EAttribute / EReference`

对每个 `textX class`，脚本遍历其 `_tx_attrs`（每个都是 `MetaAttr`）：

- `attr.ref == False`：这是值属性 → `EAttribute`
- `attr.ref == True`：这是对象引用 → `EReference`

关键映射规则：

- `containment = attr.cont`  
  - 例如 `Contract.sections` 在语法上是包含（section 属于 contract），因此 `containment="true"`
  - 例如 `Message.sender` 是引用 participant，不是包含，因此 `containment="false"`
- `eType`  
  - 基础类型：`ID/STRING → EString`，`INT → EInt/EIntegerObject`，`BOOL → EBoolean/EBooleanObject`
  - 枚举类型：`ElementState/ScalarType/... → 对应 EEnum`
  - 引用类型：`EReference` 的 `eType=#//TargetEClass`

### 1.6 多重性（lowerBound/upperBound）与 `OPTIONAL_BOUNDS`

这里要特别说明：**textX 的 `_tx_attrs` 多重性不总能反映“语法可选性”**。

例如：

```text
Message:
  'message' ... '{'
    ('initial' 'state' initialState=ElementState)?
    ('schema' schema=STRING)?
  '}'
;
```

尽管语法里是 `?`，textX 的元信息里该属性的 `mult` 可能仍显示为 `1`，但实例对象里它可能是 `None` 或空字符串等。

为了让 Ecore 的 `lowerBound` 更贴近语法真实含义、也更便于 OCL 表达，脚本用 `OPTIONAL_BOUNDS` 做“语义修正”：

- 把很多 `1..*` 修成 `0..*`（section 列表、attributes、mappings 等）
- 把 `('xxx')?` 的属性修成 `0..1`（如 `Message.initialState/schema`、`Gateway.gatewayType/initialState` 等）
- 对 `LiteralExpr` 的三个 value 字段修成 `0..1`，体现“互斥选择”的语义

当 `lowerBound==0` 且类型是 `INT/BOOL` 时，脚本倾向于用对象类型：

- `INT`：`EIntegerObject`（可空）  
- `BOOL`：`EBooleanObject`（可空）

而像 `Participant.isMulti` 这种“最好别出现 undefined”的标志位，脚本保留为 `EBoolean` 并设 `lowerBound=1`（减少 OCL 中的 `undefined` 传播问题）。

### 1.7 产物特征与限制

- 这份 `b2c.ecore` 是为了 **OCL 校验链路**服务的“工程化 Ecore”，不是 textX 元模型的 100% 等价镜像。
- 如果 `b2c.tx` 发生变化（新增规则/字段、修改可选性），需要：
  1) 重新运行 `export_b2c_ecore.py`
  2) 视情况更新 `ENUM_RULES` / `OPTIONAL_BOUNDS`

## 2) `b2c_to_xmi.py`：把 `.b2c` 实例转换为 `.xmi`

### 2.1 输入/输出

- 输入：
  - `b2c.tx`：用于 textX 解析 `.b2c`
  - `b2c.ecore`：用于 PyEcore 创建 `EObject` 并决定如何序列化
  - `*.b2c`：实例文本（如 `build/b2c/chaincode.b2c`）
- 输出：
  - `*.xmi`：EMF 实例模型（默认 `MDAcheck/chaincode.xmi`）

运行：

```bash
./.venv/bin/python MDAcheck/b2c_to_xmi.py \
  --in build/b2c/chaincode.b2c \
  --out MDAcheck/chaincode.xmi
```

### 2.2 核心难点：跨引用要保持“对象身份一致”

`.b2c` 里大量使用引用：

- `Message.sender` / `Message.receiver` 引用 `Participant`
- `FlowTarget.target` 引用 `Message/Gateway/...`
- `ParamMapping.globalRef` 引用 `GlobalVar`

在 XMI 里，这些引用必须指向同一个 `EObject` 实例，否则会出现：

- 引用丢失
- 产生重复对象（同名但不是同一对象）

因此脚本采用 **两遍（two-pass）** 方式：

#### Pass 1：创建所有“包含树”中的 EObjects

- textX 模型里，包含关系通过元信息 `MetaAttr.ref == True && MetaAttr.cont == True` 体现。
- `_iter_contained_textx_objects()` 从 `tx_model` 出发，沿着所有 containment 引用遍历，拿到“实例树”中的每个对象。
- 对每个 textX 对象，根据其 `__class__.__name__` 找到 Ecore 里的同名 `EClass`，创建 `ecls()` 得到空 `EObject`，并保存到 `tx_to_eobj[id(tx_obj)]`。

这样做可以保证：后面任何地方引用同一个 textX 对象时，都能拿到同一个 `EObject`。

#### Pass 2：填充属性与引用

对每个 textX 对象：

- 遍历 `eobj.eClass.eAllStructuralFeatures()`（包含继承来的特征）
- 如果对应字段是 `EAttribute`：
  - 普通类型：直接 `setattr(eobj, fname, value)`
  - `EEnum`：用 `_enum_literal()` 把 `READY/INACTIVE/...` 这类字符串变成 `EEnumLiteral` 再赋值
- 如果对应字段是 `EReference`：
  - containment：把子对象 `EObject` 放到列表或单值中
  - non-containment：把引用目标映射到对应 `EObject`（可能在其他 section/位置）

最后把 `Model` 作为根对象写入一个 XMI Resource 并保存。

### 2.3 为什么需要加载 `b2c.ecore`（而不是直接把 textX 模型 dump 掉）

原因是：Eclipse/EMF OCL 的校验对象必须是 **基于 Ecore 类型系统的 EObject**。

脚本通过：

- `ResourceSet().get_resource(URI(b2c.ecore))` 加载 EPackage
- `rset.metamodel_registry[pkg.nsURI] = pkg` 注册元模型

这样保存 XMI 时会带上：

- `xmlns:b2c="https://chaincollab/newTranslator/b2c"`
- `xsi:type="b2c:ParticipantSection"` 这种类型信息

Eclipse 打开 XMI 时才能正确关联到 `.ecore` 并解析类型/引用。

### 2.4 可选字段：为什么脚本要“看原始文本关键字”

textX 对“可选字段”的默认值行为比较复杂：

- `Message.initialState` 这种可选字段在实例中可能是 `None`
- `Participant.msp` 这类可选字段在实例中可能会被初始化为空字符串 `''`
- `Participant.multiMin/multiMax` 可能默认 `0`

如果直接把这些默认值都写进 XMI，会导致：

- 在 EMF/OCL 侧看起来“字段被显式设置了”，与 `.b2c` 原文语义不一致
- 某些 OCL 约束（例如用 `oclIsUndefined()` 表达“没配置”）失去意义

为此脚本引入了基于原文片段的判断：

- textX 每个对象都带 `_tx_position/_tx_position_end`，能定位它在源文本中的片段
- `_span_text()` 取出该对象对应的文本范围
- `OPTIONAL_ATTRIBUTE_KEYWORDS` 定义“某个字段的关键字模式”
  - 例如 `Participant.msp` 对应 `\bmsp\b`
  - 例如 `Message.initialState` 对应 `\binitial\s+state\b`
- 如果该字段在 Ecore 中 `lowerBound==0` 且对象片段里找不到关键字，就不把它写入 XMI

这个机制的效果是：**只有当 DSL 文本里真的出现了该字段，XMI 才写这个字段**。

注意：这是一种工程化权衡（基于关键字的启发式），如果未来 DSL 语法变化，需要同步更新这些关键字。

### 2.5 产物特征与限制

- 当前脚本把 “文本实例 → XMI 实例” 做成了结构与引用的一次映射；它不是通用编译器，也不会推导业务语义。
- 若你修改了 `b2c.ecore` 的可选性（`lowerBound`），会影响哪些字段会被“按关键字过滤”。
- 若 `.b2c` 出现了 Ecore 中没有的类名/字段名，会报错（这是预期：元模型和实例必须一致）。

---

## 3) 一条完整链路（建议工作流）

1. 修改了 `b2c.tx` 后：
   - 先跑 `export_b2c_ecore.py` 更新 `b2c.ecore`
2. 有一个 `.b2c` 模型要校验：
   - 跑 `b2c_to_xmi.py` 生成 `.xmi`
3. 在 Eclipse 中：
   - 导入/打开 `b2c.ecore`
   - 加载 `check.ocl`
   - 打开 `.xmi`，执行 OCL 校验

---

## 4) 维护指南（当 DSL 演进时怎么改脚本）

### 4.1 `export_b2c_ecore.py`

- 新增/修改枚举：更新 `ENUM_RULES`
- 调整字段可选性/多重性：更新 `OPTIONAL_BOUNDS`
- 新增 match rule 且字面量不是合法标识符：不要直接做 `EEnum`，优先转 `EString` 或做“符号→安全名字”的映射

### 4.2 `b2c_to_xmi.py`

- 新增可选字段：如果你希望“只有写了关键字才落盘”，在 `OPTIONAL_ATTRIBUTE_KEYWORDS` 里加一条
- 如果某字段在 `.b2c` 原文中没有关键字但仍希望落盘（例如默认值也有意义），就不要把它加到关键字表里
